## 适用场景速查

| 场景 | 推荐方案 |
|------|---------|
| 有 Python 历史包袱、长期战略、追求内存安全 | **A** |
| 全新项目无 Python 包袱、追求脚本层轻量 | B |
| 资源紧、6 个月内必须交付 | **C** |
| 团队 Go 强、可接受策划工作流变更 | D |
| 团队 Go 强、需要 Lua 脚本层、可接受双 GC | E |
| 想长期演进，先稳后大重写 | **C → A**（先方案 C，1-2 年后启动方案 A） |

## 共同的 KBEngine 设计前提（5 方案都保留）

- 多进程分布式架构（Loginapp/Baseapp/Cellapp/Dbmgr/Mgrs/Machine/Centermgr）
- 实体双端模型（Base + Cell）
- Ghost / Witness / AOI 系统
- 客户端 SDK 协议 v1 字节级兼容（确保 Unity/UE4/JS SDK 零修改）
- Manager/Worker 动态负载均衡

## 共同的非目标（5 方案都不追求）

- 运行时热补丁（参考 v3.0 选型文档，不是硬需求）
- 完全重新设计客户端协议（v2 是可选项，不是必选）
- 一步到位（所有方案都采用分阶段路线图）

## 如何选择？

1. 先回答三个问题：
   - 业务方能给你多少时间？（< 6 月 → C；> 18 月 → A/B/D/E）
   - 团队主语言？（C++ → C/A；Go → D/E；混合 → A）
   - 现有项目有多少 Python 资产？（多 → A/C；少或无 → B/D/E）

2. 用上面"决策矩阵"对你最在意的 3 个维度打分，加权求和

3. 把候选方案的"风险登记册"读一遍，问"我的团队能不能扛住这些风险"

4. 如果还在犹豫 A vs C：
   - 看团队是否能扛 Phase 0（3 月 Rust 学习）
   - 如果能 → A
   - 如果不能 → C，1-2 年后再启动 A
