# 方案 E：Go + Lua

> 状态：折中备选 (★★★☆☆) | 日期：2026-06-09
> 引擎层：Go 1.22+ | 脚本层：Lua（gopher-lua 或 CGO LuaJIT） | 替代对象：KBEngine v1.2.2
>
> **适用场景**：团队 Go 经验强，可接受双 GC 不确定性；希望获得 Lua 的开发效率但又不愿做大型 Rust 重写

---

## 一、技术方案

### 1.1 设计目标

| 目标 | 量化指标 |
|------|---------|
| 开发效率 | Go 引擎层开发周期相比 C++ 缩短 30% |
| 脚本灵活性 | 保留 Lua 热加载、动态数值调整能力 |
| 双 GC 可控 | Go GC + Lua GC 叠加 pause P99 < 15 ms |
| 协议兼容 | 同其他方案，客户端 SDK 零修改 |
| 部署 | 单二进制（含嵌入式 Lua），跨平台 |

### 1.2 与方案 D 的差异

新增一个 Lua 脚本层，承担：
- 游戏逻辑（替代方案 D 中 Go 实现的业务代码）
- 配置 + 数值 + 行为树
- 热加载（脚本级别）

### 1.3 Lua 嵌入方案对比

| 选项 | 性能 | CGO 开销 | 部署 | 推荐 |
|------|------|---------|------|------|
| `gopher-lua` (纯 Go) | 中（解释器，比 C Lua 慢 2-5x） | 无 | 简单（无 CGO） | ✅ 首选 |
| CGO + LuaJIT | 高（JIT 优化后接近 C） | 有（每次调用 ~100ns） | 复杂（CGO + JIT 平台限制） | △ 高性能场景 |
| `pure-lua via wasm` (wazero) | 中等 | 无 | 简单 | △ 实验性 |

**推荐 gopher-lua**：纯 Go 实现，CI/部署最简单；性能对游戏逻辑足够；后期热路径可改 LuaJIT。

### 1.5 关键设计决策

**D-1 LuaState 拓扑**
- 每个 Cellapp/Baseapp 进程一个 LuaState
- 所有 Entity 共享 state；逻辑隔离用模块（`require`）
- 不使用 multi-state（避免跨 state 通信复杂度）

**D-2 Go↔Lua 绑定**
- 用 gopher-lua 的 `LFunction` / `LUserData` 暴露 Entity
- 自动生成绑定层：从 entitydef YAML/JSON 生成 Go 注册代码
- 每次调用开销 ~500ns（gopher-lua 解释器），比 mlua 慢 10x，比 pyo3 慢 2-3x

**D-3 双 GC 治理**
- Go GC：调优 GOGC + `sync.Pool` 减少分配
- Lua GC：增量模式 + 控制单步步长（`collectgarbage("setstepmul", ...)`）
- 关键：避免在 Lua 中产生大量临时对象，Go 侧池化共享对象
- 监控：分别记录 Go pause 和 Lua pause，独立 alarm

**D-4 实体模型**
- Go 侧 Entity 是 struct，承载属性序列化、网络同步、AOI
- Lua 侧 Entity 是 metatable，承载用户逻辑（onTimer, onMove, onChat 等）
- 关键路径（移动同步、AOI 更新）保留在 Go，不进 Lua

**D-5 协议兼容**
- 同其他方案：v1 字节级兼容 + v2 protobuf

**D-6 热更新**
- 开发期：保存 .lua 文件 → 触发 `LuaState.DoFile()` → 替换 module table
- 生产期：仍然推荐停服更新；但热补丁可用于紧急 bug 修复

### 1.6 风险登记册

| # | 风险 | 概率 | 影响 | 缓解 |
|---|------|------|------|------|
| R1 | 双 GC 叠加导致延迟不可预测 | 高 | 高 | 持续 profiling；脚本层避免大对象；必要时切 LuaJIT |
| R2 | gopher-lua 性能不足 | 中 | 中 | 热路径改 Go；极端场景换 CGO LuaJIT |
| R3 | 现有 Python 项目迁移 | 极高 | 极高 | 提供 py2lua 工具（同方案 B） |
| R4 | 实体逻辑 Go/Lua 分裂，开发者认知负担 | 中 | 中 | 严格规范"什么放 Go 什么放 Lua" |
| R5 | gopher-lua 维护活跃度 | 低 | 中 | 已是 Go 生态事实标准，社区稳定 |

## 二、开发计划

| Phase | 主题 | 人月 | 团队规模 | 日历 |

### 2.2 Phase 0 — 设计 + 原型 (2 个月)

- 月 1：架构设计；评估 gopher-lua 性能基线
  - 基线测试：1000 entity，每实体 onTimer 100ms 一次
  - 验收：Go GC P99 < 3ms + Lua pause P99 < 5ms
- 月 2：原型 — Go Actor + gopher-lua Entity 绑定
  - 验收：在 Lua 中定义 Avatar，Go 调用 onMove 工作正常
- **Go/No-Go**：性能达标 + 团队能驾驭 Go+Lua → 进入 Phase 1

### 2.3 Phase 1 — 基础设施 (4-6 个月)

与方案 D 一致：协议层、网络层、数据库、日志、服务发现。

### 2.4 Phase 2 — 框架 + Lua 绑定 + py2lua (5-7 个月)

| Sprint | 内容 |
|--------|------|
| S1-2 | 实体框架（Actor + Mailbox） |
| S3-4 | EntityDef YAML/JSON 定义系统 |
| S5-8 | gopher-lua 嵌入 + Entity UserData + KBEngine.* API（30 个） |
| S9-10 | 热加载机制 |
| S11-14 | py2lua 工具开发（同方案 B 的迁移工具，可复用大部分） |

**里程碑 M2**：能用 Lua 定义 Entity；py2lua 转换 KBEngine demo

### 2.5 Phase 3 — 核心组件 (7-10 个月)

与方案 D 一致，但 Cellapp/Baseapp 暴露给 Lua 而非自带逻辑。

### 2.6 Phase 4 — 工具链 + 压测 + 迁移文档 (4-6 个月)

- Logger / Bots / Interfaces / WebConsole
- 客户端 SDK 适配
- 单元测试 + 集成测试 + 压测（重点关注双 GC 行为）
- **迁移文档**：Python → Lua 指南 + 双 GC 调优指南

### 2.8 成功标准

- [ ] 协议 v1 兼容
- [ ] 单 Cellapp ≥ 3000 实体 @ 10 Hz
- [ ] Go pause + Lua pause 叠加 P99 < 15 ms
- [ ] py2lua 工具对真实项目转换率 ≥ 60%
- [ ] 至少 1 个 demo 项目完成迁移

### 2.9 与 方案 A/B 的取舍

| 维度 | 方案 E | 方案 A/B |
|------|--------|---------|
| 周期 | 18-24 月 | 21-28 月 |
| 内存安全 | Go 弱保证（仍有 data race 可能） | Rust 编译期保证 |
| 性能确定性 | 双 GC 不可预测 | 零 GC，可预测 |
| 开发效率 | Go 学习快（1-2 周） | Rust 学习慢（2-4 月） |
| 招聘 | Go 人才市场充足 | Rust 人才稀缺 |
| 行业验证 | MMOG 领域案例少 | Rust 案例增多但仍少 |

### 2.10 长期风险

- 双 GC 是本方案最大的技术债。3 年后若线上出现"特定玩法触发 GC 风暴"的事故，可能需要紧急切换 LuaJIT 甚至重写到 Rust。
- 建议从 Phase 1 起就建立完善的 GC 监控基础设施，避免"上线后才发现"。

---

> **决策建议**：选择本方案的条件：
> 1. 团队 Go 强、Rust 弱，且不愿做 3 月学习投入
> 2. 接受双 GC 带来的延迟不确定性
> 3. 可接受 Lua 重写现有 Python 项目
> 4. 业务对 P99 延迟容忍度 ≥ 15 ms
