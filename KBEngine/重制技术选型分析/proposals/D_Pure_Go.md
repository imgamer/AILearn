# 方案 D：纯 Go 重写（无脚本层）

> 状态：不推荐 (★★☆☆☆) — 仅在特定场景下评估 | 日期：2026-06-09
> 引擎层：Go 1.22+ | 脚本层：无（游戏逻辑也用 Go 编写） | 替代对象：KBEngine v1.2.2
>
> **适用场景**：游戏品类不依赖运维期快速数值/玩法迭代；团队完全 Go 化；可接受停服更新

---

## 一、技术方案

### 1.1 设计目标与诚实声明

**设计目标**

| 目标 | 量化指标 |
|------|---------|
| 开发效率 | 单功能开发周期相比 C++ 缩短 30%+ |
| 工程化 | go mod 统一依赖，gofmt 强制风格 |
| 并发模型 | goroutine + channel 替代手写 epoll |
| 部署 | 单二进制部署，跨平台直接编译 |
| GC 延迟 | Tick 周期 100ms 内 GC pause < 5% |

**诚实声明：本方案与 KBEngine 设计哲学冲突**

KBEngine 的核心假设是"C++ 引擎 + Python 脚本"。本方案放弃脚本层，意味着：

1. **失去开发期热加载**：策划改个掉落表都要重新编译、重启
2. **失去快速迭代**：游戏逻辑改动周期从秒级（reload）变为分钟级（编译+重启）
3. **失去脚本生态**：没有 Python/Lua 帮你处理 JSON/HTTP/数据驱动配置，所有都自己写
4. **失去策划友好**：策划不可能学 Go，必须开发"配置驱动"框架来弥补

如果以上任一点是项目硬需求，**应转向方案 E (Go + Lua) 或方案 A/B**。

### 1.2 子系统技术选型

| KBEngine 子系统 | Go 替代 | 备注 |
|----------------|---------|------|
| 网络层 | `net` + `gnet` v2 或自实现 epoll wrapper | gnet 提供事件驱动 + 多 reactor |
| 协程/并发 | goroutine + channel | 天然适配 Actor 模型 |
| 序列化 | protobuf（推荐）+ `encoding/gob` 备选 | 工业标准 |
| MySQL/Redis/MongoDB | `database/sql` + `go-redis/v9` + `mongo-driver` | 成熟 |
| 导航 (Recast/Detour) | CGO 绑定 | 注意 CGO 开销 |
| 数学 | `gonum` + 自实现 Vec3/Quat | gonum 偏科学计算，游戏数学要自补 |
| 服务发现 | etcd / consul / nats.io | 替代 Machine UDP |
| 日志 | `log/slog`（标准库）+ `lumberjack` 滚动 | Go 1.21+ |
| 配置 | `viper` 或自定义 YAML/JSON | 失去 Python 灵活性需补强 |
| 进程间通信 | gRPC（推荐）或 NATS | |
| 实体模型 | 结构体 + 接口 + entity-system 库（如 `donburi`） | 无现成成熟方案，可能自研 |
| 游戏逻辑 | Go 编写（无脚本层） | **本方案与其他方案的核心差异** |

### 1.3 关键设计决策

**D-1 无脚本层的工程化补偿**
- 引入"配置驱动"框架：YAML/JSON/TOML 描述属性、技能、AI 行为树
- 数值/玩法调整通过修改配置 + 配置热加载完成（不重启进程）
- 复杂逻辑必须编译进二进制；提供"行为树/状态机"DSL 减少代码量
- 副作用：策划不能直接写脚本，必须由程序员协助

**D-2 GC 控制策略**
- 实体使用值类型 + `sync.Pool` 减少分配
- AOI 计算结果复用切片，避免每 Tick 分配
- 关键路径用 `runtime.GC()` 显式控制（不推荐，仅在万不得已）
- GOGC 调优 + 监控 `runtime.MemStats.PauseNs` P99

**D-3 实体模型**
- 没有 KBEngine 那样的 ScriptObject 双端模型
- 采用 Actor 模型：每实体一个 goroutine + mailbox channel
- Base/Cell 分离仍然保留，但实现完全用 Go 重写
- AOI：每 Cellapp 单独 goroutine 周期性计算视野差分

**D-4 协议兼容**
- 同方案 A/B：`kb-proto` 等价 Go package 实现 v1 协议字节级兼容
- 新协议默认 protobuf

**D-5 热更新/不停机更新**
- 不支持运行时热替换（Go plugin 仅限 Linux，ABI 脆弱）
- 通过"蓝绿部署 + 状态迁移"实现不停机：
  - 启动新版本进程
  - 新连接路由到新进程
  - 老连接由旧进程消化完毕
  - Cellapp 状态通过"Spawn 新 Cellapp + 实体迁移"完成
- 实现复杂度高，是本方案最大的工程债

### 1.4 风险登记册

| # | 风险 | 概率 | 影响 | 缓解 |
|---|------|------|------|------|
| R1 | 失去脚本层导致策划/数值同事工作受阻 | 极高 | 极高 | 强制开发"配置 + 行为树 DSL"框架，工作量与脚本层相当 |
| R2 | GC 在大规模实体场景下不可预测 | 中 | 高 | 持续 profiling，对象池化，必要时降级到值类型 ECS |
| R3 | 现有 Python 项目无法迁移 | 极高 | 极高 | 没有缓解，本方案承认重新开发游戏逻辑 |
| R4 | CGO Recast 调用频繁导致性能下降 | 中 | 中 | 批量化调用，Cellapp Tick 内集中 |
| R5 | Go 在 MMOG 领域案例少，未知坑多 | 中 | 中 | 与 Go 游戏服务器厂商交流（如腾讯、网易内部 Go 网关团队） |
| R6 | 不停机部署系统复杂度高 | 高 | 高 | 接受"停服维护"作为默认运维策略 |

### 2.2 Phase 0 — 设计 + 原型 (2 个月)

- 月 1：架构设计文档；评审"无脚本层"对业务的影响
  - 必须输出：策划工作流改造方案
  - 必须输出：配置驱动框架的 API 草案
- 月 2：原型 — 在 Go 中实现 Actor 模型 + AOI 简化版
  - 验证：1000 实体 @ 10 Hz 下 GC pause P99 < 5 ms
- **Go/No-Go 评审**：业务方接受"无脚本层"工作流 + 性能原型达标 → 进入 Phase 1

### 2.3 Phase 1 — 基础设施 (4-6 个月)

| Sprint | 内容 |
|--------|------|
| S1-2 | Go 模块结构 + lint/CI/构建矩阵 |
| S3-4 | 协议层（KBE v1 字节级兼容 + protobuf v2） |
| S5-6 | 网络层（gnet TCP + UDP + WebSocket） |
| S7-8 | 数据库层（MySQL/Redis/MongoDB） |
| S9-10 | 日志/监控（slog + Prometheus + OpenTelemetry） |
| S11-12 | 服务发现（etcd 客户端封装） |

**里程碑 M1**：Unity 客户端可与新服务端完成 v1 协议握手

### 2.4 Phase 2 — 服务器框架 + 配置驱动 DSL (5-7 个月)

| Sprint | 内容 |
|--------|------|
| S1-4 | 实体框架（Actor 模型 + Mailbox channel + 生命周期） |
| S5-6 | EntityDef（替代 XML，用 Go struct + tag） |
| S7-10 | **配置驱动 DSL**：属性表、技能表、AI 行为树解释器 |
| S11-12 | 配置热加载（不重启进程） |
| S13-14 | Base/Cell 分离 + 跨进程 Mailbox |

**里程碑 M2**：能用 Go struct 定义 Entity，配置热加载工作

### 2.5 Phase 3 — 核心组件 (7-10 个月)

| 子项目 | 人月 |

**里程碑 M3**：完成一个 Go demo 游戏（替代 ouroboros 的 Go 版本），端到端登录、移动、AOI

### 2.6 Phase 4 — 不停机部署系统 + 工具链 + 压测 (4-6 个月)

- 不停机部署框架（蓝绿 + 状态迁移），最难的部分
- Logger / Bots / Interfaces / WebConsole
- 客户端 SDK 适配
- 单元测试 + 集成测试 + 压测
- **必须的额外文档**：从"Python 写逻辑"到"Go + 配置"的开发指南

**里程碑 M4 (Alpha)**：完成不停机部署演示；至少 1 个 demo 项目在生产环境跑

### 2.8 成功标准

- [ ] 客户端 SDK v1 协议兼容
- [ ] 单 Cellapp ≥ 3000 实体 @ 10 Hz（GC 影响下，比 A/B 略低预期）
- [ ] GC pause P99 < 10 ms
- [ ] 配置驱动 DSL 覆盖 80% 常规数值/技能/AI 需求
- [ ] 不停机部署系统通过 24 小时压测
- [ ] 至少 1 个 Go demo 项目完整上线

### 2.9 后路：转向方案 E（Go + Lua）

如果 Phase 0/1 末发现"无脚本层"对业务影响过大（极有可能），可在 Phase 2 时嵌入 Lua（gopher-lua 或 CGO LuaJIT），转向方案 E。增量工作量约 30-40 人月。
