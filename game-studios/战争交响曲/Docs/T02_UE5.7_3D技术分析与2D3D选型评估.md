## 一、3D方案技术概览

### 1.1 项目基本信息

| 项目属性 | 值 |
|----------|-----|
| 项目类型 | 3D回合制战术RPG手游（F2P） |
| 引擎版本 | Unreal Engine 5.7 |
| 核心渲染方案 | 3D Skeletal Mesh + Nanite + Lumen（可选） + 俯视/等距相机 |
| 参考产品 | 梦幻模拟战手游（3D化）、火焰纹章Engage（3D战场）、皇家骑士团Reborn（3D战场） |
| 目标平台 | iOS / Android / PC（跨平台） |
| 网络架构 | 客户端-服务端，服务端权威 |
| 目标帧率 | 移动端30fps / PC 60fps |
| 屏幕方向 | 横屏（Landscape） |
| 画面风格 | 3D半写实/风格化，等距俯视视角 |

```

### 1.3 核心技术决策理由（3D方案）

| 决策 | 理由 |
|------|------|
| 选择3D Skeletal Mesh而非Paper2D | 骨骼动画流畅可混合、3D战场高低差表现、战斗特写震撼、UE5的强项是3D |
| Nanite用于静态建筑/地形 | 无需手动LOD、自动简化、大量节省美术工时、PCG配合生成环境 |
| Lumen可选启用 | 动态光照无需烘焙、天气变化实时反映、但移动端需降级 |
| 等距俯视相机 | 保持战术RPG经典视角、兼顾3D深度表现、触屏友好 |
| 8套基础骨骼覆盖54种职业 | 骨骼复用大幅减少动画制作量、通过材质/附件/缩放区分外观 |
| Substrate Materials | UE5.7 Production Ready、角色材质表现力提升、移动端可降级到传统管线 |

## 二、3D角色系统

### 2.1 骨骼网格体（Skeletal Mesh）

#### 2.1.1 骨骼分类与复用策略

54种职业不需要54套独立骨骼。通过骨骼复用策略，仅需8-10套基础骨骼即可覆盖所有职业：

| 骨骼编号 | 骨骼名称 | 覆盖职业 | 骨骼特点 | 预估骨骼数 |
|----------|----------|----------|----------|-----------|
| SK_Humanoid_Light | 人形轻装骨骼 | 轻步兵/剑客/法师/弓手/治疗师 | 标准人形、轻甲、灵活 | ~65 bones |
| SK_Humanoid_Heavy | 人形重装骨骼 | 重步兵/重甲骑士/指挥官 | 标准人形、重甲、宽肩 | ~70 bones |
| SK_Cavalry_Light | 轻骑兵骨骼 | 轻骑兵/游侠 | 人形+坐骑一体化 | ~90 bones |
| SK_Cavalry_Heavy | 重骑兵骨骼 | 重骑兵/圣骑士 | 人形+坐骑一体化、重甲 | ~95 bones |
| SK_DragonRider | 龙骑骨骼 | 龙骑士 | 人形+龙坐骑 | ~120 bones |
| SK_Flying_Pegasus | 飞马骨骼 | 飞马骑士 | 人形+飞马 | ~110 bones |
| SK_Dragon | 龙骨骼 | 火龙/冰龙/雷龙 | 四足+翅膀 | ~85 bones |
| SK_Siege | 攻城骨骼 | 炮兵/投石车 | 机械结构 | ~40 bones |
| SK_Boss_Large | 大型Boss骨骼 | 巨型Boss | 超大体型 | ~100 bones |
| SK_Special | 特殊骨骼 | 史莱姆/幽灵等 | 按需定制 | ~30 bones |

**骨骼复用区分外观的方式**：

| 区分方式 | 说明 | 示例 |
|----------|------|------|
| 材质替换 | 同骨骼不同材质/纹理 | 轻步兵用布料材质，法师用法袍材质 |
| 附件系统 | 通过Skeletal Mesh Socket挂载不同武器/装备 | 剑客挂载剑，弓手挂载弓 |
| 缩放调整 | 调整骨骼缩放改变体型 | 重甲骑士肩宽放大1.2倍 |
| Morph Target | 变形目标微调面部/体型 | 不同英雄的面部差异 |
| 材质实例 | 同材质不同参数的材质实例 | 红色法师袍/蓝色法师袍 |

#### 2.1.2 Animation Blueprint设计

每个基础骨骼对应一个Animation Blueprint，共8个AnimBP：

**通用状态机结构**：

```
                    ┌──────────┐
          ┌────────→│  Idle    │←────────┐
          │         └────┬─────┘         │
          │              │ MoveStart     │ MoveEnd/Stop
          │              ↓               │
          │         ┌──────────┐         │
          │    ┌───→│  Walk    │─────────┘
          │    │    └────┬─────┘
          │    │         │ Speed>RunThreshold
          │    │         ↓
          │    │    ┌──────────┐
          │    │    │  Run     │─────────┐
          │    │    └────┬─────┘         │
          │    │         │ AttackTrigger │ MoveEnd
          │    │         ↓               │
          │    │    ┌──────────┐         │
          │    │    │ Attack   │─────────┘
          │    │    └────┬─────┘
          │    │         │ HitTrigger
          │    │         ↓
          │    │    ┌──────────┐
          │    │    │   Hit    │──→ Idle
          │    │    └────┬─────┘
          │    │         │ DeathTrigger
          │    │         ↓
          │    │    ┌──────────┐
          │    └────│  Death   │ (终态)
          │         └──────────┘
          │
          │  (Cast → Idle, Heal → Idle)
          └────────────────────────────────
```

**AnimBP资产命名**：

```

ABP_3D_HumanoidHeavy    // 人形重装AnimBP
ABP_3D_CavalryLight     // 轻骑兵AnimBP
ABP_3D_CavalryHeavy     // 重骑兵AnimBP
ABP_3D_DragonRider      // 龙骑AnimBP
ABP_3D_FlyingPegasus    // 飞马AnimBP
ABP_3D_Dragon           // 龙AnimBP
ABP_3D_Siege            // 攻城AnimBP
```

#### 2.1.3 动画序列清单

每个基础骨骼需要8-10个Animation Sequence：

| 动画名称 | 类型 | 帧数/时长 | 循环 | 说明 |
|----------|------|----------|------|------|
| Anim_Idle | 待机 | 60帧(2s@30fps) | 是 | 呼吸/轻微晃动 |
| Anim_Walk | 行走 | 30帧(1s@30fps) | 是 | 战场移动 |
| Anim_Run | 跑步 | 24帧(0.8s@30fps) | 是 | 骑兵冲锋 |
| Anim_Attack_Melee | 近战攻击 | 30帧(1s@30fps) | 否 | 含AnimNotify命中帧 |
| Anim_Attack_Ranged | 远程攻击 | 30帧(1s@30fps) | 否 | 含AnimNotify发射帧 |
| Anim_Hit | 受击 | 18帧(0.6s@30fps) | 否 | 含AnimNotify受击帧 |
| Anim_Death | 死亡 | 45帧(1.5s@30fps) | 否 | 含AnimNotify淡出帧 |
| Anim_Cast | 施法 | 36帧(1.2s@30fps) | 否 | 法师/治疗师专用 |
| Anim_Heal | 治疗 | 36帧(1.2s@30fps) | 否 | 治疗师专用 |
| Anim_Special | 特殊技能 | 45帧(1.5s@30fps) | 否 | 职业特殊技能 |

**总动画量计算**：

```
8套骨骼 × 10个动画 = 80个Animation Sequence
vs 2D方案的 54职业 × 5动画 = 270个Flipbook

3D方案动画资产量仅为2D方案的30%
但每个3D动画的制作工时远高于2D帧动画
```

#### 2.1.5 Animation Montage设计

战斗动画序列使用Montage组织，支持精确的段落控制：

```
Montage_AM_AttackMelee:
  Section: Start → WindUp → Swing → Hit → Recovery
  AnimNotify位置:
    - WindUp开始: AN_CameraShake(Light)
    - Swing开始: AN_WeaponSwing
    - Hit帧: AN_AttackHit + AN_CameraShake(Heavy)
    - Recovery: 无

Montage_AM_CastSpell:
  Section: Start → Channel → Release → Recovery
  AnimNotify位置:
    - Channel开始: AN_CastEffect(蓄力特效)
    - Release帧: AN_CastEffect(释放特效) + AN_CameraShake(Medium)
    - Recovery: 无

Montage_AM_CriticalHit:
  Section: Start → WindUp → Swing → Impact → SlowMo → Recovery
  AnimNotify位置:
    - Swing: AN_WeaponSwing + AN_SpawnTrail
    - Impact: AN_AttackHit + AN_CameraShake(Extreme)
    - SlowMo: AN_SlowMotion(0.3x, 0.5s)
```

### 2.2 角色LOD与Nanite

#### 2.2.1 Nanite限制

**关键限制**：Nanite不支持Skeletal Mesh动画。Nanite仅能用于静态网格（Static Mesh），如建筑、地形装饰物等。

| 资产类型 | 是否支持Nanite | 说明 |
|----------|---------------|------|
| 角色Skeletal Mesh | ❌ 不支持 | 骨骼动画与Nanite不兼容 |
| 武器Static Mesh | ✅ 支持 | 但通常作为附件挂载，不单独用Nanite |
| 建筑Static Mesh | ✅ 支持 | Nanite最佳应用场景 |
| 地形装饰物 | ✅ 支持 | 岩石/树木等 |
| 地形Landscape | ❌ 不适用 | Landscape有自己的LOD系统 |

#### 2.2.2 角色手动LOD方案

角色需要手动制作3级LOD：

| LOD级别 | 三角面数 | 渲染距离 | 骨骼数 | 说明 |
|---------|---------|---------|--------|------|
| LOD0 | 8000-12000 | 0-15m | 全部(~65) | 近距离完整模型 |
| LOD1 | 3000-5000 | 15-30m | 简化(~30) | 中距离简化模型 |
| LOD2 | 800-1500 | 30-60m | 极简(~15) | 远距离低模 |
| Imposter | 2(Quad) | 60m+ | 0 | Billboard贴图替代 |

**LOD设置**：

```
Skeletal Mesh Editor → LOD Settings:
  - LOD0: Source Model (完整模型)
  - LOD1: Reduction Settings
    - Triangle Ratio: 0.4
    - Bone Removal: Remove bones below importance threshold
  - LOD2: Reduction Settings
    - Triangle Ratio: 0.12
    - Bone Removal: Aggressive removal
  - Imposter: Billboard at max distance
    - Capture angles: 8 (8方向Billboard)
    - Resolution: 256×256
```

**战场LOD距离计算**：

```
等距俯视相机(45°) + 缩放范围(300-1200):
  - 最大缩放(近距离): 角色占屏幕大 → LOD0
  - 中等缩放: 角色占屏幕中 → LOD1
  - 最小缩放(远距离): 角色占屏幕小 → LOD2/Imposter

战场20个小队同时渲染:
  - 选中/交战小队: LOD0 (2-4个)
  - 附近小队: LOD1 (6-8个)
  - 远处小队: LOD2 (8-12个)
  - 超远小队: Imposter (0-4个)
```

### 2.3 角色材质与Substrate

#### 2.3.1 UE5.7 Substrate Materials

UE5.7中Substrate Materials已达到Production Ready状态，是新一代材质/光照系统：

| 特性 | 传统材质 | Substrate材质 |
|------|---------|--------------|
| 着色模型 | 固定选择(Default Lit/Subsurface等) | 自由组合SL(Substrate Layer) |
| 多层材质 | 有限支持 | 原生支持多层混合 |
| 物理准确性 | 近似 | 更物理准确 |
| 性能 | 优化成熟 | 略高开销(但UE5.7已优化) |
| 移动端 | 完全支持 | UE5.7支持，可降级到传统 |

#### 2.3.2 角色材质方案

| 材质类型 | Substrate方案 | 移动端降级方案 | 使用场景 |
|----------|--------------|---------------|---------|
| 皮肤 | SL_Dielectric + SSS | Traditional Subsurface Profile | 人形角色面部/手部 |
| 金属铠甲 | SL_Metal + ClearCoat | Traditional Default Lit + Metal | 重甲骑士/圣骑士 |
| 布料 | SL_Dielectric + Fuzz | Traditional Subsurface + Cloth | 法师袍/轻步兵服装 |
| 皮革 | SL_Dielectric | Traditional Default Lit | 靴子/皮带 |
| 头发 | SL_Dielectric + SSS + TangentNormal | Traditional Hair | 英雄头发 |
| 龙鳞 | SL_Metal + Iridescence | Traditional Default Lit + Pearlscent | 龙族单位 |

**材质实例策略**：

```
Master Material: MM_Character_Substrate (PC/主机)
  ├── MI_Warrior_Light    (轻步兵：布料+皮革)
  ├── MI_Warrior_Heavy    (重步兵：金属+皮革)
  ├── MI_Mage_Robe        (法师：布料+发光)
  ├── MI_Archer_Leather   (弓手：皮革+布料)
  ├── MI_Cavalry_Armor    (骑兵：金属+布料)
  └── MI_Dragon_Scale     (龙：金属+虹彩)

Master Material: MM_Character_Mobile (移动端降级)
  ├── (同上，使用传统着色模型)
```

**角色材质参数**：

```
MM_Character_Substrate:
  - Base Color Texture: T_[Character]_BC
  - Normal Map: T_[Character]_N
  - ORM Map: T_[Character]_ORM (Occlusion/Roughness/Metallic)
  - Emissive Mask: T_[Character]_E (法师发光部位)
  - Substrate Layers:
    - Layer 1: Base Material (Dielectric/Metal)
    - Layer 2: SSS (皮肤区域)
    - Layer 3: ClearCoat (铠甲区域)
  - Parameters:
    - TintColor: 阵营颜色(红/蓝)
    - EmissiveStrength: 发光强度(法师)
    - RoughnessOverride: 粗糙度覆盖
    - MetallicOverride: 金属度覆盖
```

#### 2.3.3 移动端材质降级策略

```
移动端材质降级流程:
  1. 项目设置 → Rendering → Substrate → Enable Substrate: true
  2. Device Profile → Android_High → Substrate: true
  3. Device Profile → Android_Mid → Substrate: false (降级到传统)
  4. Device Profile → Android_Low → Substrate: false + 简化材质

移动端简化材质:
  - 去除SSS → 使用SimpleSubsurfaceProfile
  - 去除ClearCoat → 使用Default Lit
  - 去除Fuzz → 使用Default Lit
  - 减少纹理采样: BC+N → 仅BC
  - 降低纹理分辨率: 2048→1024→512
```

## 三、3D战场环境

### 3.1 地形系统

#### 3.1.1 Landscape战场地形

3D方案使用Landscape替代2D方案的TileMap，实现真正的3D地形：

| 属性 | 2D方案(TileMap) | 3D方案(Landscape) |
|------|----------------|-------------------|
| 地形表示 | 平面Tile拼接 | 高度图+3D网格 |
| 高低差 | 数据标记(0/1/2) | 真实3D高度 |
| 视觉效果 | 像素Tile | 3D地形+材质 |
| 编辑方式 | TileMap编辑器 | Landscape编辑器 |
| 地形过渡 | 手动选择变体 | 自动材质混合 |

**Landscape参数**：

```
战场Landscape设置:
  - 尺寸: 40×30格 (对应40×30的战场网格)
  - 每格世界单位: 128uu × 128uu
  - 总世界尺寸: 5120uu × 3840uu
  - 高度图分辨率: 401×301 (每格1个采样点)
  - 最大高度差: 512uu (4格高度差)
  - LOD: 4级 (Landscape自带LOD)
  - Component数量: 4×3 = 12个Component
  - Subsection: 1×1 (每Component 1个Subsection)
```

**高度图与地形属性映射**：

```
高度值 → 地形属性:
  - 0-50uu: 低地(平地) → Terrain.Lowland
  - 50-200uu: 中地(丘陵) → Terrain.Hills
  - 200-512uu: 高地(山地) → Terrain.Highland

高度差影响:
  - 攻击方在高地: ATK+10%, 射程+1
  - 防御方在高地: 被命中率-10%
  - 移动消耗: 每上升1级高度, 移动消耗+1
```

#### 3.1.2 地形材质

8种地形类型通过Landscape Layer混合实现：

| Layer名称 | 材质 | 物理效果 | 游戏属性 |
|-----------|------|---------|---------|
| Grass | 草地材质+草丛细节 | 脚步声-草地 | 移动消耗1, 无掩护 |
| Forest | 泥土材质+树根细节 | 脚步声-泥土 | 移动消耗2, 闪避+20% |
| Mountain | 岩石材质+碎石细节 | 脚步声-岩石 | 移动消耗3, 掩护+1 |
| Sand | 沙地材质+沙粒细节 | 脚步声-沙地 | 移动消耗2, 骑兵减速 |
| Water | 水面材质+水波细节 | 脚步声-水 | 移动消耗3, 被雷克制 |
| Road | 石板路材质 | 脚步声-石板 | 移动消耗0.5, 无掩护 |
| Wall | 石墙材质 | - | 不可通行, 掩护+2 |
| Bridge | 木桥材质 | 脚步声-木头 | 移动消耗1, 无掩护 |

**Landscape材质节点图**：

```
ML_LandscapeTerrain:
  - 8个Landscape Layer Blend节点
  - 每层:
    - Base Color Texture (Tiling: 4×4)
    - Normal Map
    - Roughness Map
    - Displacement Map (可选, PC端)
  - 自动混合: Layer Blend by Painted Weight
  - 全局参数:
    - SnowAmount (暴风雪覆盖)
    - Wetness (雨天湿润)
    - HeatDistortion (热浪扭曲)
```

#### 3.1.3 网格叠加

3D地形上需要叠加半透明网格显示格子：

```
Grid Mesh Overlay:
  - 类型: Plane Mesh (40×30格)
  - 材质: M_GridOverlay (半透明+Unlit)
  - 渲染: 在Landscape上方10uu处

void ABP_Pawn_BattleCamera_3D::FollowSquad(ABP_Character_Squad_3D* TargetSquad)
{
    if (!TargetSquad) return;
    
    FVector TargetLocation = TargetSquad->GetActorLocation();
    FVector CurrentLocation = GetActorLocation();
    
    float LerpSpeed = 5.0f;
    FVector NewLocation = FMath::Lerp(CurrentLocation, TargetLocation, LerpSpeed * GetWorld()->GetDeltaSeconds());
    SetActorLocation(NewLocation);
}

### 7.2 寻路与移动

#### 7.2.1 A*寻路

3D地形上的网格寻路需要考虑高度差：

```cpp
TArray<FIntPoint> UGridManager::FindPath(FIntPoint Start, FIntPoint End, int32 MaxMoveRange)
{
    TMap<FIntPoint, float> GCost;
    TMap<FIntPoint, FIntPoint> CameFrom;
    TPriorityQueue<FIntPoint> OpenSet;

    GCost.Add(Start, 0.f);
    OpenSet.Push(Start, 0.f);

    while (!OpenSet.IsEmpty())
    {
        FIntPoint Current = OpenSet.Pop();

        if (Current == End)
        {
            return ReconstructPath(CameFrom, End);
        }

        for (FIntPoint Neighbor : GetNeighbors(Current))
        {
            if (!IsPassable(Neighbor)) continue;

            float HeightDiff = FMath::Abs(GetCell(Neighbor).Height - GetCell(Current).Height);
            if (HeightDiff > MaxClimableHeight) continue;

            float MoveCost = GetCell(Neighbor).MoveCost;
            if (HeightDiff > 0) MoveCost += HeightDiff * HeightMovePenalty;

            float TentativeG = GCost[Current] + MoveCost;

            if (!GCost.Contains(Neighbor) || TentativeG < GCost[Neighbor])
            {
                GCost.Add(Neighbor, TentativeG);
                CameFrom.Add(Neighbor, Current);
                float F = TentativeG + Heuristic(Neighbor, End);
                OpenSet.Push(Neighbor, F);
            }
        }
    }

    return TArray<FIntPoint>();
}
```

## 十三、混合方案：2.5D

### 13.1 2.5D方案描述

2.5D是2D和3D的混合方案，参考八方旅人(Octopath Traveler)和三角战略(Triangle Strategy)的成功案例：

| 层 | 渲染方式 | 内容 | 技术 |
|----|----------|------|------|
| 战场环境 | 3D | Nanite建筑+Landscape地形+PCG植被 | UE5 3D全功能 |
| 角色渲染 | 2D | Sprite/Flipbook立于3D场景中 | Paper2D+PaperZD |
| 特效 | 3D | Niagara 3D特效 | UE5 Niagara |
| UI | 2D | UMG/CommonUI | UE5 UMG |

**参考产品分析**：

| 产品 | 2.5D实现 | 视觉效果 | 商业表现 |
|------|---------|---------|---------|
| 八方旅人 | 2D角色+3D场景 | HD-2D风格，独特美感 | 全球销量300万+ |
| 三角战略 | 2D角色+3D场景 | HD-2D风格，战术RPG | 全球销量100万+ |
| 梦幻模拟战(3D化) | 3D角色+3D场景 | 全3D，画质提升 | 国服收入大幅增长 |

### 13.2 2.5D技术要点

#### 13.2.1 Sprite在3D场景中的渲染

```
Paper2D Sprite Component可存在于3D世界中:
  - UPaperFlipbookComponent → 放置在3D场景中
  - Sprite始终面向相机(Billboard)
  - Sprite位置由Grid Manager的3D坐标决定
  - Sprite的Y轴排序实现前后遮挡

设置:
  - Sprite Component → Attach to Scene
  - Billboard: Always Face Camera
  - Sort Rule: By Y Position (3D空间Y轴排序)
  - Render in 3D Space: true
```

#### 13.2.2 光照处理

```
2D角色在3D场景中的光照挑战:
  - 2D Sprite无法接收3D光照
  - 2D Sprite无法投射3D阴影
  - 2D角色与3D环境光照不一致

解决方案:
  - 2D角色: Unlit材质(自发光) + 环境光颜色参数
  - 天气变化: 通过材质参数调整角色颜色(偏蓝/偏暖)
  - 3D场景: 正常使用Lumen/Baked光照
  - 光照一致性: 通过Post Process统一色调

角色材质:
  M_Sprite_2_5D:
    - Shading Model: Unlit
    - Base: SpriteTextureSampler → Emissive
    - Tint: World Ambient Color (从场景获取)
    - Weather: 天气颜色叠加
    - Flash: 受击闪白
```

#### 13.2.3 阴影处理

```
2D角色阴影:
  - 不使用3D阴影(2D Sprite无法投射)
  - 使用Blob Shadow(圆形阴影贴图)
  - Blob Shadow Component: 放置在角色脚下
  - 颜色: 半透明黑色
  - 大小: 根据角色高度缩放
  - 形状: 椭圆形(模拟透视)

Blob Shadow实现:
  - Decal Component 或 Plane Mesh + 半透明材质
  - 位置: 角色正下方(Landscape表面)
  - 旋转: 贴合地形法线
  - 透明度: 根据角色高度衰减
```

#### 13.2.4 相机设置

```
2.5D相机:
  - 等距俯视: 45° Pitch
  - 2D角色始终面向相机(Billboard)
  - 相机旋转: 限制范围(避免2D角色侧面)
  - 相机缩放: 可缩放但2D角色不缩放(保持像素清晰)

相机限制:
  - Yaw旋转: 限制±30°(避免2D角色侧面)
  - Pitch: 固定-45°到-60°
  - 缩放: 仅调整SpringArm长度
```

#### 13.2.5 战斗演出

```
2.5D战斗演出:
  - 相机特写: 可推进到近景(与3D相同)
  - 2D角色: 播放Flipbook攻击动画
  - 3D特效: Niagara 3D特效(与3D相同)
  - 相机震动: 支持(与3D相同)
  - 慢动作: 支持(与3D相同)
  - 限制: 2D角色无法做3D旋转/位移动画
```

### 13.3 2.5D评分

| 维度 | 2.5D评分 | 说明 |
|------|---------|------|
| 视觉表现力 | 8 | 3D战场氛围+2D角色风格化，HD-2D独特美感 |
| 移动端性能 | 7 | 3D场景消耗GPU，但2D角色消耗低，总体中等 |
| 开发效率 | 7 | 3D场景+2D角色，效率介于2D和3D之间 |
| 战场表现 | 8 | 3D地形+高低差，与纯3D相同 |
| 战斗演出 | 6 | 3D特效+相机，但2D角色动画有限 |
| 美术成本 | 6 | 3D场景成本+2D角色成本，总成本中等 |
| 技术风险 | 7 | 2D/3D混合的技术挑战(光照/阴影一致性) |
| 商业化 | 7 | HD-2D风格有独特受众，但付费天花板低于纯3D |
| 市场差异 | 8 | HD-2D风格在手游市场差异化明显 |
| **综合** | **7.1** | 折中方案，各维度均衡 |

### 14.2 最终推荐

#### 14.2.1 推荐方案：3D方案（附条件）

**推荐3D方案**，理由如下：

**1. UE5的强项是3D，Paper2D是弱项**

```
UE5引擎能力对比:
  3D能力: ★★★★★ (Nanite/Lumen/MegaLights/PCG/Substrate/Niagara)
  2D能力: ★★☆☆☆ (Paper2D维护少/无AnimBP/无合批/社区弱)

选择UE5做2D项目=用F1赛车跑越野
选择UE5做3D项目=用F1赛车跑赛道
```

**2. 市场差异化：3D战术RPG是蓝海**

```
手游市场现状:
  - 像素风RPG: 红海(大量竞品)
  - 2D战术RPG: 红海(梦幻模拟战/天地劫/铃兰之剑)
  - 3D战术RPG: 蓝海(梦幻模拟战3D化后竞品少)
  
3D战术RPG的差异化:
  - 3D战场高低差/深度感
  - 3D战斗特写/相机控制
  - 3D皮肤/特效付费
  - 画质党+泛用户拓展
```

**3. 3D付费天花板更高**

```
3D皮肤定价空间:
  - 2D换色皮肤: ¥6-12
  - 2D新立绘皮肤: ¥18-30
  - 3D新模型皮肤: ¥30-68
  - 3D特效皮肤(含专属特效): ¥68-128

LTV对比:
  - 2D战术RPG: 月ARPU ¥30-50
  - 3D战术RPG: 月ARPU ¥50-100
  
3D方案多出的¥1.63M/年人力成本，可通过更高的ARPU回收
```

**4. 骨骼复用策略降低长线成本**

```
54种职业的骨骼复用:
  - 首次制作: 8套骨骼+54个Mesh (成本高)
  - 新增职业: 复用骨骼+新Mesh+材质变体 (成本低)
  - 新增皮肤: 材质实例+附件替换 (成本最低)

vs 2D:
  - 首次制作: 54职业×5动画=270 Flipbook (成本中)
  - 新增职业: 全新Flipbook (成本中)
  - 新增皮肤: 换色/新立绘 (成本中-高)

长线运营3年:
  - 2D: 每年新增10职业 = 50 Flipbook/年
  - 3D: 每年新增10职业 = 复用骨骼+10 Mesh/年
  - 3D长线成本更低
```

#### 14.2.2 关键前提条件

3D方案需要满足以下前提条件，否则建议退回2.5D或2D方案：

| 前提条件 | 最低要求 | 验证方式 |
|----------|---------|---------|
| 团队3D美术 | ≥4人(建模+动画+特效+TA) | 招聘/外包评估 |
| UE5 3D经验 | 至少1人有UE5 3D手游项目经验 | 面试/作品集 |
| 预算 | 年预算≥¥700万 | 财务评估 |
| 目标机型 | 中高端机(iPhone 12+/骁龙888+) | 市场调研 |
| 性能验证 | 中端机≥30fps | 早期性能测试 |

#### 14.2.3 风险缓解

```
3D方案风险缓解策略:

1. 移动端性能风险:
   - 第2周: 搭建3D性能测试场景(20个角色+建筑+特效)
   - 第4周: 在中端机上测试，确认≥30fps
   - 如果<25fps: 降级到2.5D方案

2. 美术产能风险:
   - Phase 1(4周): 制作1套骨骼+3个职业Mesh+动画
   - 验证制作管线和时间估算
   - 如果超时>30%: 考虑外包部分美术

3. 包体风险:
   - 第8周: 打包测试，确认首包≤2GB
   - 如果>2GB: 启用更激进的资源压缩+按需下载

4. 降级方案:
   - 如果3D方案无法满足性能/包体/工期:
     → 降级到2.5D方案(保留3D场景+2D角色)
     → 再降级到2D方案(Paper2D+PaperZD)
   - GAS逻辑层完全复用，降级成本可控
```

#### 14.2.4 分阶段实施建议

```
Phase 0: 技术验证(4周)
  - 搭建3D性能测试场景
  - 制作1套骨骼+3个职业
  - 验证移动端性能
  - 验证美术管线
  - Go/No-Go决策

Phase 1: 核心战斗(12周)
  - 3D战场+网格系统
  - 4套核心骨骼+20个职业
  - 战斗特写+相机系统
  - GAS集成

Phase 2: 养成系统(8周)
  - 养成/抽卡/商店等(与2D相同)

Phase 3: 美术量产(20周)
  - 剩余34个职业
  - 30个战场
  - 特效量产

Phase 4: 优化(8周)
  - 移动端性能优化
  - 包体优化
  - 内存优化

总计: 52周(约13个月)
```

## 附录A：2D/3D方案切换指南

由于GAS逻辑层在2D和3D方案中完全相同，项目可以在2D和3D之间切换：

```
切换步骤:
1. 替换角色渲染组件:
   - 2D: UPaperFlipbookComponent → PaperZD AnimBP
   - 3D: USkeletalMeshComponent → AnimBP + Montage

2. 替换战场渲染:
   - 2D: TileMap + Sprite
   - 3D: Landscape + Nanite Mesh + PCG

3. 替换相机:
   - 2D: 固定俯视相机
   - 3D: SpringArm + 可旋转/缩放/特写

4. 替换特效:
   - 2D: NS_2D_* (Sprite Renderer)
   - 3D: NS_3D_* (Mesh+Sprite+Ribbon+Light)

5. GAS逻辑层: 无需修改
   - AttributeSet: 相同
   - GameplayEffect: 相同
   - GameplayTag: 相同
   - MMC/Execution: 相同

6. 数据层: 无需修改
   - DataTable: 相同
   - DataAsset: 相同
   - Data Registry: 相同

7. UI层: 无需修改
   - Widget: 相同
   - CommonUI: 相同

建议:
  - GA基类(C++): 定义通用逻辑，与渲染无关
  - GA子类(蓝图): 分别实现2D/3D的表现层
  - 通过Feature Switch或配置切换2D/3D模式
```
