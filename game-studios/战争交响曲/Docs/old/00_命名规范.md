# 文档命名规范

## 格式

```
YYYYMMDD_vX.X_文档类型_项目名称.md
```

或对于细分模块：

```
YYYYMMDD_vX.X_文档类型_模块名称.md
```

## 字段说明

| 字段 | 说明 | 示例 |
|------|------|------|
| YYYYMMDD | 创建日期，8位数字 | `20260319` |
| vX.X | 版本号，主版本.次版本 | `v1.0` |
| 文档类型 | 文档分类关键词 | `玩法分析`, `核心系统需求`, `UI原型`, `战斗流程图` |
| 模块名称 | 具体模块或项目名 | `战争交响曲尼菲林传奇`, `小队编制系统` |

## 示例

✅ **正确示例：**
```
20260319_v1.0_玩法分析_战争交响曲尼菲林传奇.md
20260319_v1.0_核心系统需求_战争交响曲尼菲林传奇.md
20260315_v0.1_主界面原型_主菜单.png
20260310_v1.0_战斗流程_战斗系统.vsdx
20260301_v1.0_需求评审会议纪要_第一次会议.md
```

❌ **错误示例：**
```
需求文档.md                  (缺少日期和版本)
核心系统.md                  (不规范)
20260319_需求.md            (缺少版本号)
v1.0_需求文档.md            (缺少日期)
我的设计思路.txt             (格式不对)
```

## 版本号规则

| 版本号 | 说明 |
|--------|------|
| v0.1 | 初稿，草稿阶段，未评审 |
| v1.0 | 正式版，已评审，可用于开发 |
| v1.1 | 小改动，部分调整，不影响整体结构 |
| v2.0 | 大改版，结构变化大，需要重新评审 |

## 文件格式要求

| 文档类型 | 推荐格式 | 说明 |
|----------|----------|------|
| 需求文档 | Markdown (`.md`) | 纯文本，Git友好，易于阅读 |
| 流程图 | Draw.io (`.drawio`) 或 Visio (`.vsdx`) | 源文件保存方便编辑 |
| UI原型 | Figma (链接) 或 PNG | 图片导出方便预览 |
| 会议纪要 | Markdown (`.md`) | 简单清晰 |
| 参考素材 | 保留原格式 | JPG/PNG/PDF |

## 目录分类对应关系

| 目录 | 文档类型关键词 |
|------|----------------|
| 01_requirements | 玩法分析, 需求, 规格, 规则 |
| 02_prototype | 原型, UI, 界面, 草图 |
| 03_flowchart | 流程, 流程图, 状态图 |
| 04_meeting | 会议纪要, 讨论记录 |
| 05_assets | 参考, 素材, 原图, 截图 |
| 06_review | 评审, 测试报告, 验收 |

## 维护原则

1. **同一文档更新时，新版本覆盖旧位置，更新版本号**
   - 不要保留多个版本在同一目录，Git已经记录了历史
   - 如果需要保留历史版本，可以打Git标签

2. **新增分类直接增加序号目录即可**
   - 当前 `01`~`06`，后续新增直接 `07_xxx`
   - 保持序号连续递增

3. **所有文档都要在 README.md 索引表中登记**
   - 新增文档后更新索引表格
   - 便于快速检索

## Git相关

- 所有文本文件（*.md）正常提交Git
- 大文件二进制文件建议使用Git LFS或存在外部
- .gitignore 已经排除临时文件和大文件
