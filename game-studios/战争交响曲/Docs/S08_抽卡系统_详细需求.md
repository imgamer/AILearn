# S08 抽卡系统 - 详细需求文档

## 1. 系统概述

抽卡系统是玩家获取英雄、佣兵、神器的核心途径。系统包含3个独立卡池：

| 卡池 | 主要产出 | 货币 | 保底机制 |
|------|---------|------|---------|
| 英雄池 | 英雄碎片/整卡 | 钻石 | 90抽SSR保底，180抽大保底 |
| 佣兵池 | 佣兵契约 | 金币 | 30抽SR保底 |
| 神器池 | 神器装备 | 钻石 | 30抽紫保底 |

- 英雄池和神器池使用钻石，佣兵池使用金币
- 英雄池和神器池支持UP池轮换，每2周更换UP内容
- 佣兵池为常驻池，无UP机制

## 2. 数据结构

### 2.1 GachaPool - 卡池

```csharp
public class GachaPool
{
    public string pool_id;              // 卡池ID，如 "hero_001"
    public PoolType pool_type;          // 卡池类型枚举: hero / mercenary / artifact
    public string name;                 // 卡池名称，如 "英雄UP池·烈焰骑士"
    public int cost_per_pull;           // 单抽消耗货币数量
    public int cost_per_ten;            // 十连消耗货币数量（通常=单抽×10，可配置折扣）
    public float ssr_rate;              // SSR概率（英雄池专用）
    public float sr_rate;               // SR概率
    public float r_rate;                // R概率
    public float n_rate;                // N概率
    public int pity_threshold;          // 小保底阈值（触发必出高稀有度的抽数）
    public int big_pity_threshold;      // 大保底阈值（触发必出UP的抽数）
    public int current_pity_count;      // 当前保底计数（玩家维度，非池维度）
    public List<string> up_items;       // 当前UP物品ID列表
}
```

### 2.2 GachaResult - 抽卡结果

```csharp
public class GachaResult
{
    public ItemType item_type;          // 物品类型枚举: hero / mercenary / artifact
    public string item_id;              // 物品ID
    public Quality quality;             // 品质枚举: SSR / SR / R / N（英雄池）
                                        // 或 Orange / Purple / Blue / Green（神器池）
    public bool is_new;                 // 是否新获得（英雄维度去重判断）
    public bool is_up;                  // 是否为UP物品
}
```

### 2.3 GachaHistory - 抽卡记录

```csharp
public class GachaHistory
{
    public string pull_id;              // 抽卡唯一ID，UUID格式
    public string pool_id;              // 所属卡池ID
    public long timestamp;              // 抽卡时间戳（Unix秒）
    public List<GachaResult> results;   // 本次抽卡结果列表（单抽为1个，十连为10个）
    public int total_cost;              // 本次总消耗
}
```

## 4. 英雄池概率

### 4.1 基础概率

| 品质 | 概率 | 说明 |
|------|------|------|
| SSR | 2% | 其中UP占SSR的50%，即总体1%出UP-SSR |
| SR | 10% | 其中UP各占SR的2.5%（3个UP-SR共7.5%） |
| R | 30% | 均匀分配 |
| N | 58% | 均匀分配 |

### 4.2 概率计算伪代码

```
function determineQuality(pool, pityCount, bigPityTriggered):
    // 大保底：必出UP-SSR
    if bigPityTriggered:
        return Quality.SSR, is_up=true

    // 小保底：必出SSR
    if pityCount >= pool.pity_threshold:
        // 小保底SSR中50%概率为UP
        if random(0,1) <= 0.5:
            return Quality.SSR, is_up=true
        else:
            return Quality.SSR, is_up=false

    // 正常概率
    roll = random(0, 1)
    if roll < pool.ssr_rate:          // 0 ~ 0.02
        // SSR中50%为UP
        if random(0,1) <= 0.5:
            return Quality.SSR, is_up=true
        else:
            return Quality.SSR, is_up=false
    elif roll < pool.ssr_rate + pool.sr_rate:  // 0.02 ~ 0.12
        // SR中UP各占2.5%
        up_roll = random(0,1)
        if up_roll < 0.075:  // 3个UP-SR共7.5%
            return Quality.SR, is_up=true
        else:
            return Quality.SR, is_up=false
    elif roll < pool.ssr_rate + pool.sr_rate + pool.r_rate:  // 0.12 ~ 0.42
        return Quality.R, is_up=false
    else:
        return Quality.N, is_up=false
```

### 4.3 保底机制

| 保底类型 | 触发条件 | 效果 | 计数重置 |
|---------|---------|------|---------|
| 小保底 | 连续89抽未出SSR | 第90抽必出SSR | 出SSR后重置为0 |
| 大保底 | 连续179抽未出UP-SSR | 第180抽必出UP-SSR | 出UP-SSR后重置为0 |
| 十连保底 | 十连前9抽均为N | 第10抽必出SR及以上 | 不影响pity计数 |

**保底计数规则：**
- `current_pity_count`：每抽+1，出SSR时重置为0
- `big_pity_count`（隐含）：每抽+1，出UP-SSR时重置为0；出非UP-SSR时不重置
- 十连保底独立判定，不影响pity计数

### 4.4 软保底（可选）

从第75抽开始，SSR概率线性提升：
```
actual_ssr_rate = ssr_rate + (pityCount - 74) * 0.06
// 第75抽: 2% + 6% = 8%
// 第80抽: 2% + 36% = 38%
// 第89抽: 2% + 90% = 92%
// 第90抽: 100%（硬保底）
```

## 7. 抽卡流程

### 7.1 单抽流程

```
PullSingle(pool_id, player_id):
    1. 获取卡池配置 pool = TB_GachaPool[pool_id]
    2. 获取玩家数据 player = GetPlayer(player_id)
    3. 检查货币是否足够
       IF player.currency[pool.cost_currency_type] < pool.cost_per_pull:
           RETURN Error("货币不足")
    4. 扣除货币
       player.currency[pool.cost_currency_type] -= pool.cost_per_pull
    5. 增加pity计数
       player.pity_count[pool_id] += 1
       player.big_pity_count[pool_id] += 1
    6. 判定是否触发保底
       big_pity_triggered = (player.big_pity_count[pool_id] >= pool.big_pity_threshold)
       pity_triggered = (player.pity_count[pool_id] >= pool.pity_threshold)
    7. 按权重随机抽取品质
       quality, is_up = determineQuality(pool, player.pity_count[pool_id], big_pity_triggered)
    8. 在该品质内按权重随机抽取具体物品
       item = weightedRandomSelect(TB_GachaItem where pool_id && quality)
       IF is_up:
           item = weightedRandomSelect(TB_GachaItem where pool_id && quality && is_up)
    9. 检查是否重复（英雄池）
       is_new = !player.owned_heroes.Contains(item.item_id)
    10. 重复英雄转化为碎片
        IF NOT is_new AND item.item_type == hero:
            actual_reward = Fragment(item.fragment_id, item.fragment_count)
        ELSE:
            actual_reward = item
            player.owned_heroes.Add(item.item_id)
    11. 重置保底计数
        IF quality >= SSR (英雄池) OR quality >= SR (佣兵池) OR quality >= Purple (神器池):
            player.pity_count[pool_id] = 0
        IF is_up AND quality == SSR (英雄池):
            player.big_pity_count[pool_id] = 0
    12. 记录历史
        history = GachaHistory(pull_id=UUID(), pool_id, timestamp=Now(), results=[result], total_cost=pool.cost_per_pull)
        player.gacha_history.Append(history)
    13. 播放动画（客户端）
        RETURN GachaResult(item_type, item_id, quality, is_new, is_up)
```

### 7.2 十连流程

```
PullTen(pool_id, player_id):
    1. 获取卡池配置 pool = TB_GachaPool[pool_id]
    2. 获取玩家数据 player = GetPlayer(player_id)
    3. 检查货币是否足够
       IF player.currency[pool.cost_currency_type] < pool.cost_per_ten:
           RETURN Error("货币不足")
    4. 扣除货币
       player.currency[pool.cost_currency_type] -= pool.cost_per_ten
    5. 执行10次单抽逻辑（步骤5~11），收集结果到 results[]
    6. 十连保底检查
       has_sr_or_above = results.Any(r => r.quality >= SR)  // 英雄池
       IF NOT has_sr_or_above:
           // 将最后一个结果替换为SR及以上
           results[9] = forceMinSR(pool_id)
    7. 记录历史
       history = GachaHistory(pull_id=UUID(), pool_id, timestamp=Now(), results, total_cost=pool.cost_per_ten)
       player.gacha_history.Append(history)
    8. 按品质排序结果（SSR在前，N在后），用于动画展示
       results.Sort((a,b) => a.quality.CompareTo(b.quality))
    9. 播放动画（客户端）
       RETURN results
```

## 8. UP池轮换规则

### 8.1 轮换周期

- 每2周（14天）更换一次UP内容
- 更换时间：每周一 00:00:00（服务器时间）
- 排期由 TB_GachaSchedule 配置驱动

### 8.2 轮换内容

| 轮换项 | 说明 |
|--------|------|
| UP英雄 | 1个SSR英雄，从英雄池SSR列表中轮换 |
| UP-SR英雄 | 3个SR英雄，从英雄池SR列表中轮换 |
| UP神器 | 1个橙色神器（神器池轮换） |

### 8.3 轮换逻辑

```
function refreshUpPool():
    current_schedule = TB_GachaSchedule.Where(s =>
        s.start_time <= Now() AND s.end_time > Now()
    )
    FOR EACH schedule IN current_schedule:
        pool = TB_GachaPool[schedule.pool_id]
        pool.up_items.Clear()
        pool.up_items.Add(schedule.up_hero_id)
        FOR EACH sr_id IN schedule.up_sr_ids:
            pool.up_items.Add(sr_id)
        pool.name = generatePoolName(schedule)
```

### 8.4 保底继承

- UP池轮换时，pity计数**不重置**，继承到下一个UP池
- big_pity计数**不重置**，继承到下一个UP池
- 玩家在旧池积累的保底进度在新池中继续有效

### 8.5 排期冲突处理

- 新排期开始时，旧排期自动关闭
- 如果新排期配置缺失，旧排期延长至新排期配置生效
- 不允许出现同一pool_type的多个排期同时生效

## 11. 服务器校验要点

1. 所有概率判定在服务器端执行，客户端仅负责展示
2. 抽卡请求需校验：玩家ID、卡池ID、货币数量、卡池开放状态
3. 每次抽卡操作为原子事务：货币扣除与奖励发放必须同时成功或同时失败
4. 抽卡历史至少保留90天
5. 保底计数持久化到数据库，不可丢失
6. 防并发：同一玩家同时只能有一个抽卡请求在处理中
