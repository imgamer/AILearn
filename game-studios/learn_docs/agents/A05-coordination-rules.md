## 📋 协调规则概览

在 Claude Code Game Studios 项目中，智能体协作遵循一套明确的规则体系，确保多智能体协作时的秩序和效率。

### 五大核心协调规则

```
┌─────────────────────────────────────────────────────────────┐
│                    协调规则体系                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  规则 1：垂直委派（Vertical Delegation）                   │
│  ─────────────────────────────────────────                 │
│  领导层 → 部门负责人 → 专家层                              │
│  复杂决策不跨级，逐级委派和升级                            │
│                                                             │
│  规则 2：横向咨询（Horizontal Consultation）               │
│  ─────────────────────────────────────────                 │
│  同层级可以咨询，但不越权做决策                            │
│  咨询结果仅供参考，决策权属于负责人                        │
│                                                             │
│  规则 3：冲突解决（Conflict Resolution）                   │
│  ─────────────────────────────────────────                 │
│  冲突升级到共同上级裁决                                    │
│  无共同上级时，升级到创意总监（创意冲突）                  │
│  或技术总监（技术冲突）                                    │
│                                                             │
│  规则 4：变更传播（Change Propagation）                    │
│  ─────────────────────────────────────────                 │
│  影响多领域的设计变更，由制作人协调传播                    │
│  确保所有相关方知晓并调整                                  │
│                                                             │
│  规则 5：领域边界（Domain Boundaries）                     │
│  ─────────────────────────────────────────                 │
│  智能体不得无委派修改其指定目录外的文件                    │
│  跨领域修改需获得该领域负责人的显式委派                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 规则 1：垂直委派（Vertical Delegation）

### 规则定义

```
领导层（Tier 1）→ 部门负责人（Tier 2）→ 专家层（Tier 3）

• 复杂决策不跨级
• 逐级委派和升级
• 每一级只对其直接下级负责
```

### 为什么需要垂直委派

| 问题 | 解决方案 |
|------|----------|
| 上下文过载 | 领导层不需要知道实现细节，只关注战略 |
| 决策质量 | 让最了解细节的人做具体决策 |
| 责任明确 | 清晰的汇报链，避免责任模糊 |
| 效率提升 | 减少不必要的升级，加快决策速度 |

### 正确 vs. 错误的委派方式

```markdown
❌ 错误的跨级委派：
用户：@creative-director，这个按钮用什么颜色？

Creative Director（Tier 1）：这个... 也许是蓝色？
（问题：创意总监不应该关心具体按钮颜色，
 这应该是 UI 设计师或美术总监的决定）

✅ 正确的垂直委派：
用户：@creative-director，这个按钮用什么颜色？

Creative Director（Tier 1）：颜色选择应该基于我们的 
视觉设计系统。让我委派给合适的人。

@art-director（Tier 2），请为这个项目确定按钮颜色方案。
请确保符合我们的整体视觉风格。

Art Director（Tier 2）：收到。基于我们的设计系统，
主按钮使用品牌蓝色（#0078D4），次按钮使用灰色。

@ui-programmer（Tier 3），请实现这个颜色方案。

UI Programmer（Tier 3）：已实现。所有按钮使用正确的颜色。

✅ 结果：每个层级处理合适的问题，决策质量高
```

### 升级路径

当问题超出当前层级的能力范围时，需要升级到上级：

```
升级触发条件：
• 需要更高权限的决策
• 涉及多个部门，需要协调
• 超出当前层级的专业知识
• 冲突无法在当前层级解决

升级路径：
Tier 3（专家）→ Tier 2（部门负责人）→ Tier 1（领导层）

示例：
@gameplay-programmer（Tier 3）：
"这个性能问题我优化不了，需要重构架构"

→ 升级到 @lead-programmer（Tier 2）：
"建议重构渲染管线，但会影响其他系统，需要协调"

→ 升级到 @technical-director（Tier 1）：
"批准架构重构，协调各部门配合"
```

## 规则 2：横向咨询（Horizontal Consultation）

### 规则定义

```
同层级智能体可以相互咨询，但：
• 咨询结果仅供参考，不具约束力
• 决策权属于被咨询者的直接上级或委托人
• 不得越权做出影响其他领域的决策
```

### 为什么需要横向咨询

| 好处 | 说明 |
|------|------|
| 知识共享 | 不同领域的专业知识可以相互补充 |
| 早期发现问题 | 其他视角能发现本领域忽视的问题 |
| 提高决策质量 | 基于更多信息做出更好的决策 |
| 促进协作 | 增强团队间的理解和信任 |

### 正确 vs. 错误的横向咨询

```markdown
❌ 错误的横向决策：
@game-designer（Tier 2）："@lead-programmer，这个技能的伤害公式
应该用乘法还是加法？"

@lead-programmer（Tier 2）："应该用乘法，这样后期数值更可控。"
（问题：主程序员做出了设计决策，这超出了他的职责范围。
 虽然他的建议有价值，但最终决策权应该属于游戏设计师
 或需要升级到创意总监）

✅ 正确的横向咨询：
@game-designer（Tier 2）："@lead-programmer，关于这个技能的伤害公式，
我在考虑两种方案：
A. 乘法公式：伤害 = 攻击力 × 技能系数
B. 加法公式：伤害 = 攻击力 + 技能系数 × 等级

从技术实现角度，哪种方案更容易维护？
是否会影响性能？"

@lead-programmer（Tier 2）："从实现角度：
- 两种方案在性能上没有明显差异
- 乘法方案更容易扩展（可以加入更多乘数）
- 但需要注意浮点数精度问题

建议：如果数值范围很大，考虑使用定点数或
对数空间计算来避免精度问题。"

@game-designer（Tier 2）：
"感谢建议！基于你的技术输入，我决定使用乘法方案，
并加入数值压缩机制防止精度问题。

最终方案：
伤害 = 基础攻击力 × 技能系数 × (1 + 暴击加成)
使用定点数表示，范围 0.001 ~ 999999"

✅ 结果：
- 游戏设计师做出了设计决策（正确的决策权）
- 主程序员提供了有价值的技术输入（有效的横向咨询）
- 最终决策考虑了技术约束（高质量的决策）
```

### 咨询的边界

```markdown
可以咨询的范围：
✅ 技术可行性评估
✅ 实现复杂度估算
✅ 性能影响分析
✅ 架构建议
✅ 风险评估
✅ 最佳实践分享

不可以越权的范围：
❌ 做出设计决策（应由设计师决策）
❌ 做出产品决策（应由制作人/创意总监决策）
❌ 改变项目范围（应由制作人决策）
❌ 批准预算（应由制作人/技术总监决策）
```

## 规则 3：冲突解决（Conflict Resolution）

（继续编写冲突解决、规则4、规则5的内容...）

由于篇幅限制，我将提供剩余规则的关键要点：

### 规则 3：冲突解决

- 冲突升级到共同上级裁决
- 创意冲突 → 创意总监
- 技术冲突 → 技术总监
- 制作冲突 → 制作人

### 规则 4：变更传播

- 影响多领域的设计变更，由制作人协调
- 确保所有相关方知晓并调整
- 使用变更请求（Change Request）流程

### 规则 5：领域边界

- 智能体不得无委派修改指定目录外的文件
- 跨领域修改需获得该领域负责人的显式委派
- 使用 `@domain-leader` 格式请求委派
