## 🎯 各层智能体的特点

### Tier 1: 领导层 - 战略视角

**核心特征：**
- 🎯 **决策权力**：做最终决策，解决冲突
- 🔭 **长远视角**：关注项目整体方向和愿景
- 🌐 **全局视野**：理解所有领域，看到整体图景
- 👥 **协调能力**：统筹资源，协调各部门

**模型选择：Opus**
- 需要最强的推理能力和创造力
- 处理复杂的战略决策和冲突解决

**典型工作流：**
```
接收复杂问题 → 分析全局影响 → 考虑多种方案 → 做出决策/协调资源 → 委派执行
```

**示例场景：**
- 创意总监裁决美术风格和游戏氛围的冲突
- 技术总监决定使用哪个引擎和架构方案
- 制作人协调各部门进度，解决资源冲突

---

### Tier 2: 部门负责人 - 战术视角

**核心特征：**
- 📋 **领域专业**：深耕特定领域，拥有专业深度
- 🎨 **质量把控**：确保输出符合标准，进行审查
- 👨‍🏫 **指导能力**：指导 Tier 3 专家，提供技术支持
- 🔗 **跨部门协作**：与其他部门负责人协调

**模型选择：Sonnet**
- 平衡推理能力和效率
- 处理复杂的战术决策和专业任务

**典型工作流：**
```
接收委派任务 → 分析专业需求 → 制定执行方案 → 指导 Tier 3 执行 → 审查输出质量
```

**示例场景：**
- 游戏设计师设计战斗系统机制
- 主程序员审查代码架构和进行代码审查
- 美术总监制定美术风格和审核美术资源
- QA 负责人制定测试策略和审查测试覆盖

---

### Tier 3: 专家层 - 执行视角

**核心特征：**
- 🛠️ **Hands-on**：实际执行具体任务，编写代码/设计内容
- 🔬 **深度专注**：专注于特定技术领域，拥有深厚专业知识
- ⚡ **高效执行**：快速完成任务，注重效率
- 📈 **持续改进**：在执行中学习和优化

**模型选择：Sonnet 或 Haiku**
- Tier 3 的通用专家使用 Sonnet
- 简单、重复性任务可使用 Haiku（更快更便宜）

**典型工作流：**
```
接收具体任务 → 分析技术需求 → 执行实现 → 自我检查 → 提交成果
```

**示例场景：**
- 游戏程序员实现玩家移动控制代码
- AI 程序员编写行为树和寻路算法
- 关卡设计师搭建关卡布局和白盒
- 技术美术编写着色器和特效
- QA 测试员编写测试用例和执行测试

## ✅ 本课小结

### 核心知识点

1. **三层架构的组织逻辑：**
   - Tier 1（领导层）：战略决策、愿景规划、冲突裁决
   - Tier 2（部门负责人）：战术管理、领域专业、质量把控
   - Tier 3（专家层）：执行工作、深度专业、hands-on

2. **模型选择策略：**
   - Tier 1：Opus（最强推理和创造力）
   - Tier 2：Sonnet（平衡推理和效率）
   - Tier 3：Sonnet 或 Haiku（执行效率优先）

3. **协作流程的核心原则：**
   - 垂直委派：从上到下分配任务
   - 并行执行：同层级可并行工作
   - 质量把关：上级审查下级输出
   - 结果整合：协调者统一输出

### 关键洞察

```
三层架构的价值：
• 专业化 - 每层专注于自己的领域，输出专业质量
• 可扩展性 - 可以不断添加新的专家智能体
• 可管理性 - 清晰的职责划分和升级路径
• 效率 - 并行工作和专业分工大幅提升效率
```

## 🙋 常见问题

**Q: 三层架构是固定的吗？我可以创建自己的层级吗？**

A: 三层架构是推荐的最佳实践，但不是绝对固定的。对于小型项目，可以简化为两层（领导层 + 执行层）。关键是保持清晰的职责划分和升级路径。

**Q: 一个智能体可以属于多个层级吗？**

A: 不可以。每个智能体应该明确属于一个层级。但在实际协作中，Tier 2 的智能体可能会执行一些 Tier 3 的具体工作，这属于"向下兼容"，而不是层级改变。

**Q: 如何处理跨层级的直接沟通？**

A: 原则上应该逐级沟通（垂直委派），但在紧急情况下或简单询问时，可以允许跨层级直接沟通。关键是要确保信息最终汇总到负责协调的智能体（通常是 Producer）。

**Q: 我应该从哪一层开始创建我的第一个智能体？**

A: 推荐从 Tier 3（专家层）开始：
1. 选择一个你熟悉的具体领域（如代码审查、文档编写等）
2. 创建该领域的专家智能体
3. 逐步扩展到 Tier 2（部门负责人）
4. 最后创建 Tier 1（领导层）智能体进行协调

这样可以循序渐进地建立你的智能体团队。
