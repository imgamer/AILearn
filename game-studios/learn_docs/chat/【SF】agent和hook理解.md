> ## 什么是父代理和子代理，创建方式有区别吗，claude code如何区分这两者？钩子为什么是sh文件，不是文本提示词吗，hook是claude code的机制吗，被claude code调度吗，还是大模型来调度和触发？

非常好的问题！这些都是理解Claude Code架构的核心概念。让我逐一为你详细解答。

## 一、父代理 vs 子代理 (Parent Agent vs Subagent)

### 概念定义

```
┌─────────────────────────────────────────────────────────────────┐
│                     父代理 (Parent Agent)                        │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  子代理 A (Subagent)    │  子代理 B (Subagent)           │   │
│  │  · 独立的上下文窗口      │  · 独立的上下文窗口            │   │
│  │  · 专注特定任务         │  · 专注特定任务                │   │
│  │  · 返回结构化结果       │  · 返回结构化结果              │   │
│  └─────────────────────────────────────────────────────────┘   │
│                           ▲                                      │
│                           │                                      │
│                    子代理结果汇总                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 创建方式的区别

| 特性        | 父代理      | 子代理                      |
| --------- | -------- | ------------------------ |
| **创建方式**  | 用户直接发起对话 | 通过父代理调用工具创建              |
| **上下文继承** | 从无到有     | 父代理选择性注入上下文              |
| **生命周期**  | 直到用户结束   | 任务完成自动结束                 |
| **调用方式**  | 自然对话     | 通过Claude Code的Subagent工具 |
| **结果处理**  | 直接展示给用户  | 返回给父代理继续处理               |

### 实际代码示例

假设父代理正在设计一个战斗系统，需要研究伤害计算公式：

```markdown
【父代理内部处理】

用户：帮我设计一个战斗系统的伤害计算公式

父代理分析：这是一个复杂任务，需要：
1. 研究现有游戏的伤害公式设计
2. 分析我们的游戏类型适合什么公式
3. 给出具体实现建议

→ 决定：创建研究子代理
```

父代理通过**工具调用**创建子代理：

```json
{
  "tool_name": "Subagent",
  "tool_input": {
    "agent_name": "combat-researcher",
    "task": "研究RPG游戏的伤害计算公式设计模式，重点关注：\n1. 线性公式 vs 对数公式的适用场景\n2. 暴击和闪避如何整合\n3. 攻防属性如何影响最终伤害",
    "context": "我们正在开发一个回合制RPG，角色等级1-100，需要平衡PVE战斗"
  }
}
```

**Claude Code系统处理：**

1. 创建新的上下文窗口（子代理）
2. 将父代理的`context`注入子代理
3. 子代理开始执行任务
4. 完成后，结果返回给父代理

## 二、钩子的技术本质

### 钩子为什么是.sh文件而不是文本提示词？

```
┌────────────────────────────────────────────────────────────────┐
│                    钩子 vs 提示词 的本质区别                        │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│   文本提示词 (Prompt)                Shell脚本 (.sh)              │
│   ───────────────────                ─────────────              │
│   · 给大模型的自然语言指令            · 给操作系统的可执行程序        │
│   · 用于指导AI如何响应               · 用于执行具体的系统操作         │
│   · 在AI内部执行                     · 在操作系统层面执行             │
│                                                                │
│   为什么钩子必须是.sh？                                            │
│   ─────────────────────                                          │
│   1. 钩子需要在Claude Code之外执行                                 │
│      → 访问git状态、文件系统、环境变量                               │
│                                                                    │
│   2. 钩子需要精确控制行为                                          │
│      → exit 0 (允许) / exit 2 (阻止)                               │
│      → 这不是自然语言能精确控制的                                   │
│                                                                    │
│   3. 钩子需要快速执行                                              │
│      → shell脚本轻量级，无需启动AI                                   │
│                                                                    │
└────────────────────────────────────────────────────────────────┘
```

### 钩子是Claude Code的机制吗？谁来调度？

```
┌──────────────────────────────────────────────────────────────────┐
│                    Claude Code 钩子调度架构                         │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │                    Claude Code 系统                      │   │
│   │  ┌─────────────────────────────────────────────────┐    │   │
│   │  │         Hook Engine (钩子引擎)                    │    │   │
│   │  │  ┌─────────────────────────────────────────────┐  │    │   │
│   │  │  │  1. 读取 .claude/settings.json           │  │    │   │
│   │  │  │     → 解析钩子配置和事件绑定               │  │    │   │
│   │  │  │                                            │  │    │   │
│   │  │  │  2. 监听Claude Code内部事件              │  │    │   │
│   │  │  │     → SessionStart/Stop                  │  │    │   │
│   │  │  │     → PreToolUse/PostToolUse             │  │    │   │
│   │  │  │     → SubagentStart                      │  │    │   │
│   │  │  │     → PreCompact                         │  │    │   │
│   │  │  │                                            │  │    │   │
│   │  │  │  3. 当事件触发时：                        │  │    │   │
│   │  │  │     → 查找绑定的钩子脚本                   │  │    │   │
│   │  │  │     → 传递输入数据(JSON)到stdin            │  │    │   │
│   │  │  │     → 执行shell脚本                        │  │    │   │
│   │  │  │     → 收集stdout/stderr                    │  │    │   │
│   │  │  │     → 根据exit code决定是否阻止操作           │  │    │   │
│   │  │  └─────────────────────────────────────────────┘  │    │   │
│   │  └─────────────────────────────────────────────────────┘    │   │
│   └─────────────────────────────────────────────────────────────┘   │
│                              ▲                                      │
│                              │ 触发                                  │
│   ┌──────────────────────────┴──────────────────────────────┐      │
│   │                    大模型 (Claude)                        │      │
│   │  · 生成响应                                              │      │
│   │  · 决定调用工具                                          │      │
│   │  · 这些行为触发Hook Engine                                │      │
│   └──────────────────────────────────────────────────────────┘      │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 七、总结

| 问题                    | 答案                                                               |
| --------------------- | ---------------------------------------------------------------- |
| **父代理 vs 子代理的区别**     | 父代理是用户直接对话的主会话；子代理是通过工具创建的独立上下文，用于并行/专注任务                        |
| **创建方式区别**            | 父代理自然创建；子代理通过`Subagent`工具调用，需指定agent\_name和task                  |
| **Claude Code如何区分**   | 通过工具调用链：用户消息→父代理→Subagent工具→新建上下文→子代理执行→结果返回                     |
| **钩子为什么是.sh**         | 钩子在操作系统层面执行，需要访问文件系统、git、环境变量；shell脚本是最轻量、最通用的可执行格式              |
| **钩子是Claude Code机制吗** | **是的**，钩子是Claude Code原生支持的机制，通过Hook Engine调度执行                   |
| **谁来调度触发**            | Claude Code的Hook Engine监听内部事件（SessionStart、PreToolUse等），自动触发绑定钩子 |

## 三、关键问题解答

### 1. 钩子是Claude Code的机制吗？谁调度触发？

```
┌─────────────────────────────────────────────────────────────────┐
│                    Claude Code 钩子调度机制                        │
```

### 2. 可以实现"部门间工作配合"模式吗？

```
┌─────────────────────────────────────────────────────────────────┐
│              两种架构模式的可行性分析                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  模式A：钩子触发的部门协作 (你设想的)     →  技术上可行，但不推荐    │
│  ─────────────────────────────────────                           │
│                                                                 │
│  实现方式：                                                     │
│  ┌─────────────────┐                                           │
│  │ validate-commit.sh (PreToolUse)                              │
│  │   检测到设计师Agent试图提交代码文件                            │
│  │        ↓                                                     │
│  │   写入信号文件: .claude/signals/transfer-needed              │
│  │        ↓                                                     │
│  │   输出消息: "检测到跨职责操作，已通知协调器"                  │
│  │        ↓                                                     │
│  │   exit 0 (允许提交，但标记需要审查)                          │
│  └─────────────────┘                                           │
│                     ↓                                           │
│  ┌─────────────────┐                                           │
│  │ 父Agent的下一轮 │                                           │
│  │ 检查到信号文件   │                                           │
│  │ 启动程序员Agent  │                                           │
│  │ 进行代码审查/重写 │                                           │
│  └─────────────────┘                                           │
│                                                                 │
│  ❌ 为什么不推荐？                                               │
│  · 复杂度高：需要额外的信号机制和协调逻辑                           │
│  · 延迟大：钩子执行后，需要等待下一轮对话才能处理                     │
│  · 脆弱性：信号文件可能被清理或丢失                                 │
│  · 违背"文件即状态"原则：引入了隐式的信号状态                        │
│                                                                 │
```

## 一、钩子插槽：Claude Code预定义的生命周期事件

```
┌─────────────────────────────────────────────────────────────────┐
│              Claude Code 预定义的钩子插槽（Hook Slots）              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  这些插槽是由Claude Code系统预定义的，就像一个个"钩子点"：              │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  插槽名称          触发时机           输入数据类型        │    │
│  ├─────────────────────────────────────────────────────────┤    │
│  │  SessionStart      会话开始时         无                  │    │
│  │  Stop              会话结束时         无                  │    │
│  │  PreCompact        上下文压缩前       无                  │    │
│  │  SubagentStart     子代理创建时       Agent信息(JSON)       │    │
│  ├─────────────────────────────────────────────────────────┤    │
│  │  PreToolUse        工具调用前         工具输入(JSON)        │    │
│  │  PostToolUse       工具调用后         工具输入/输出(JSON)    │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
│  这些插槽是Claude Code系统内置的，就像"插座"一样：                    │
│  · 系统定义了这些插槽的存在                                          │
│  · 每个插槽在特定时机被"激活"                                         │
│  · 你可以把脚本"插"到这些插槽上                                        │
│  · 插槽激活时，会自动执行插上的脚本                                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 二、如何指定sh文件插到哪个插槽？

```
┌─────────────────────────────────────────────────────────────────┐
│              通过 settings.json 配置钩子绑定                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  关键文件：.claude/settings.json                                  │
│  这个文件告诉Claude Code：当某个插槽激活时，执行哪些脚本               │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  配置结构：                                                │    │
│  │                                                           │    │
│  │  {                                                        │    │
│  │    "hooks": {                                             │    │
│  │      // 插槽名称: [脚本1, 脚本2, ...]                        │    │
│  │      "SessionStart": [                                    │    │
│  │        ".claude/hooks/session-start.sh",                  │    │
│  │        ".claude/hooks/detect-gaps.sh"                      │    │
│  │      ],                                                   │    │
│  │      "Stop": [                                            │    │
│  │        ".claude/hooks/session-stop.sh"                     │    │
│  │      ],                                                   │    │
│  │      "PreCompact": [                                      │    │
│  │        ".claude/hooks/pre-compact.sh"                      │    │
│  │      ],                                                   │    │
│  │      "SubagentStart": [                                   │    │
│  │        ".claude/hooks/log-agent.sh"                        │    │
│  │      ],                                                   │    │
│  │      // 带条件的插槽配置                                      │    │
│  │      "PreToolUse": {                                      │    │
│  │        // 只有调用Bash工具时才触发                             │    │
│  │        "Bash": [                                          │    │
│  │          ".claude/hooks/validate-commit.sh",               │    │
│  │          ".claude/hooks/validate-push.sh"                  │    │
│  │        ]                                                  │    │
│  │      },                                                   │    │
│  │      "PostToolUse": {                                     │    │
│  │        // Write或Edit工具调用后触发                          │    │
│  │        "Write": [                                         │    │
│  │          ".claude/hooks/validate-assets.sh"                │    │
│  │        ],                                                 │    │
│  │        "Edit": [                                          │    │
│  │          ".claude/hooks/validate-assets.sh"                │    │
│  │        ]                                                  │    │
│  │      }                                                    │    │
│  │    }                                                      │    │
│  │  }                                                        │    │
│  │                                                           │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
│  配置规则：                                                        │
│  · 插槽名称必须是系统预定义的（见上文列表）                            │
│  · 值可以是：                                                      │
│    - 字符串数组：[脚本路径1, 脚本路径2]                             │
│    - 条件对象：{ "工具名": [脚本数组], ... }                        │
│  · 多个脚本按顺序执行                                              │
│  · 脚本路径相对于项目根目录                                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 三、插槽是谁定义的？触发机制如何保证？

```
┌─────────────────────────────────────────────────────────────────┐
│              插槽定义者和触发机制详解                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  问题1：这些插槽是谁定义的？                                         │
│  ───────────────────────────                                       │
│                                                                 │
│  答案：由Claude Code系统预定义                                       │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  Anthropic（Claude Code的开发者）在系统中内置了这些钩子点：      │    │
│  │                                                           │    │
│  │  · 他们定义了插槽的名称（SessionStart, PreToolUse等）          │    │
│  │  · 他们定义了每个插槽的触发时机                                  │    │
│  │  · 他们定义了传递给脚本的数据格式（JSON结构）                     │    │
│  │  · 他们实现了钩子引擎，负责在正确时机调用脚本                     │    │
│  │                                                           │    │
│  │  用户（我们）能做的是：                                         │    │
│  │  · 编写要执行的脚本（.sh文件）                                  │    │
│  │  · 在settings.json中配置哪个脚本绑定到哪个插槽                   │    │
│  │                                                           │    │
│  │  我们不能：                                                   │    │
│  │  · 创建新的插槽（不能添加MyCustomHook）                         │    │
│  │  · 修改插槽的触发时机                                           │    │
│  │  · 修改传递给脚本的数据格式                                       │    │
│  │                                                           │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
│  ═══════════════════════════════════════════════════════        │
│                                                                 │
│  问题2：发生对应事件时肯定会触发吗？                                  │
│  ─────────────────────────────────                                 │
│                                                                 │
│  答案：是的，只要配置了就会触发                                        │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │  触发机制保证：                                             │    │
│  │                                                           │    │
│  │  1. 事件驱动架构                                            │    │
│  │     · Claude Code内部维护一个事件总线                          │    │
│  │     · 当特定事件发生时（如会话开始），事件被发布到总线            │    │
│  │     · 钩子引擎订阅了这些事件                                    │    │
│  │                                                           │    │
│  │  2. 同步执行                                                 │    │
│  │     · 当事件发生时，钩子引擎同步执行绑定的脚本                   │    │
│  │     · 脚本在事件处理流程中执行（PreToolUse可以阻塞操作）         │    │
│  │     · 脚本执行完成后，事件处理才继续                              │    │
│  │                                                           │    │
│  │  3. 配置检查                                                 │    │
│  │     · 钩子引擎检查settings.json中是否有对应事件的配置             │    │
│  │     · 如果有，按顺序执行所有绑定的脚本                            │    │
│  │     · 如果没有，事件继续正常处理（钩子不干预）                     │    │
│  │                                                           │    │
│  │  4. 触发时机（以PreToolUse为例）：                             │    │
│  │     · 用户/Agent请求调用Bash工具                                │    │
│  │     · Claude Code准备执行工具调用                                │    │
│  │     · 【触发PreToolUse钩子】                                    │    │
│  │     · 执行validate-commit.sh等脚本                               │    │
│  │     · 脚本返回exit 0 → 允许工具调用                             │    │
│  │     · 脚本返回exit 2 → 阻止工具调用，返回错误信息                  │    │
│  │     · 执行实际工具调用                                           │    │
│  │                                                           │    │
│  │  5. 可靠性保证：                                               │    │
│  │     · 钩子引擎是Claude Code核心组件，始终运行                      │    │
│  │     · 事件触发是系统级机制，不受用户代码影响                        │    │
│  │     · 脚本执行失败不会导致Claude Code崩溃                          │    │
│  │     · 脚本错误会记录但不会阻塞系统（除非exit 2）                     │    │
│  │                                                           │    │
│  └─────────────────────────────────────────────────────────┘    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 总结

| 问题                 | 答案                                      |
| ------------------ | --------------------------------------- |
| **插槽是谁定义的？**       | Claude Code系统预定义（Anthropic），用户只能使用，不能新增 |
| **如何指定sh文件插哪个插槽？** | 在`.claude/settings.json`的`hooks`配置中绑定   |
| **发生事件时肯定会触发吗？**   | 是的，只要配置了就会按顺序执行，这是系统级保证                 |
| **脚本可以阻止操作吗？**     | PreToolUse钩子可以通过`exit 2`阻止工具调用          |

这个钩子机制就像电路板上的插槽，系统定义了插槽的位置和功能，我们只需要把芯片（脚本）插到对应位置即可！
