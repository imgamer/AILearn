# 第一课：Skill 到底是什么？

> ⏱️ 预计学习时间：30分钟  
> 🎯 目标：理解 Skill 的核心概念、价值和适用场景

---

## 🎬 从一个场景开始

想象你正在用 Claude Code 开发一个游戏：

### ❌ 没有 Skill 的场景

```
你：帮我创建一个冲刺计划

Claude：好的，冲刺计划应该包含以下内容：
1. 首先我需要了解当前里程碑...
2. 然后查看之前的冲刺...
3. 扫描设计文档...
4. ...

（20分钟后）

Claude：好的，基于以上分析，这是冲刺计划的初稿：
[一个不够结构化的文档]

你：嗯，格式不太对，而且缺了风险部分...
```

**问题在哪？**
- 每次都要重新解释什么是"好的冲刺计划"
- 输出格式不一致
- 容易遗漏步骤
- 需要大量来回确认

---

### ✅ 使用 Skill 的场景

```
你：/sprint-plan new

Claude：正在创建新的冲刺计划...

✅ 已读取当前里程碑：MVP - 核心战斗系统
✅ 已分析 Sprint 3 完成度：85%（2个任务延期）
✅ 已扫描设计文档：3个特性标记为"就绪"
✅ 已检查风险登记册：2个中等风险需关注

---

📋 Sprint 4 计划已生成

目标：实现完整的连击系统和敌人AI反应

容量：
- 总天数：10天
- 缓冲（20%）：2天
- 可用：8天

Must Have：
| ID | 任务 | 负责人 | 估算 | 依赖 | 验收标准 |
|---|---|---|---|---|---|
| S4-1 | 轻攻击-重攻击输入检测 | gameplay-programmer | 2d | 无 | 输入缓冲200ms... |
| ... | ... | ... | ... | ... | ... |

[完整文档已写入：production/sprints/sprint-4-plan.md]

是否需要：
- A) 调整任务优先级
- B) 添加更多细节到某个任务
- C) 导出为其他格式
- D) 完成，开始执行
```

**差异在哪？**
- ✅ 一键触发，无需解释
- ✅ 格式一致，内容完整
- ✅ 步骤标准化，不会遗漏
- ✅ 快速迭代，聚焦决策

## 🤔 那么，Skill 到底是什么？

### 一句话定义

> **Skill 是封装了特定工作流程的可复用单元，让复杂的任务变成简单的 `/` 命令。**

### 类比理解

| 类比 | 解释 |
|------|------|
| **编程中的函数** | 像 `calculateDamage()` 封装了伤害计算逻辑 |
| **IDE 中的宏** | 像 "一键格式化代码" 封装了多步操作 |
| **游戏中的技能** | 按下快捷键，触发一套组合动作 |
| **餐厅的菜谱** | 按步骤执行，每次得到一致的成品 |

### 核心特征

```
┌─────────────────────────────────────────┐
│           SKILL = 工作流程的封装            │
├─────────────────────────────────────────┤
│                                         │
│  📥 输入（Input）                         │
│     - 用户提供的参数                        │
│     - 项目上下文                          │
│     - 当前状态                           │
│           ↓                              │
│  ⚙️ 处理（Processing）                    │
│     - 标准化步骤序列                        │
│     - 条件分支判断                         │
│     - 外部工具调用                         │
│           ↓                              │
│  📤 输出（Output）                        │
│     - 生成的文件                          │
│     - 状态更新                           │
│     - 用户反馈                           │
│                                         │
├─────────────────────────────────────────┤
│  ✨ 价值：一致性 + 效率 + 质量              │
└─────────────────────────────────────────┘
```

---

## 🎯 Skill 解决的核心问题

### 1. **一致性问题**

**没有 Skill 时：**
- Alice 写的冲刺计划有 5 个部分
- Bob 写的冲刺计划有 8 个部分
- Carol 完全忘了写风险部分

**使用 Skill 后：**
- 每个冲刺计划都有完全相同的结构
- 必填项不会被遗漏
- 新成员也能产出专业质量的文档

### 2. **效率问题**

**没有 Skill 时：**
```
用户：帮我做个代码审查
Claude：好的，我需要...
1. 首先让我读取文件内容（等待）
2. 然后检查编码标准（等待）
3. 再检查架构合规性（等待）
4. ...还有 10 个步骤
（总共 15 分钟）
```

**使用 Skill 后：**
```
用户：/code-review src/player.gd
Claude：[立即执行所有标准化步骤]
（总共 30 秒）
```

### 3. **知识沉淀问题**

**没有 Skill 时：**
- 专家的 know-how 存在于个人经验中
- 每次都要重新发明轮子
- 最佳实践无法规模化复制

**使用 Skill 后：**
- 专家知识编码在 Skill 定义中
- 任何人都可以调用专家级工作流程
- 持续改进 Skill 定义，全员受益

## 🔧 Skill 的类型

根据功能和复杂度，Skill 可以分为四种类型：

### Type 1: 单一任务 Skill（Single-Task Skill）

**特点：** 完成一个明确、原子的任务

**示例：**
- `/changelog` - 生成变更日志
- `/balance-check` - 检查游戏平衡性
- `/asset-audit` - 审计资源合规性

**结构：**
```
输入 → 单一处理步骤 → 输出
```

---

### Type 2: 工作流 Skill（Workflow Skill）

**特点：** 封装多步骤的标准化流程

**示例：**
- `/prototype` - 原型开发完整流程（设计→实现→测试→报告）
- `/sprint-plan` - 冲刺计划制定（分析→规划→文档化）
- `/code-review` - 代码审查（读取→检查→报告）

**结构：**
```
输入 → 步骤1 → 步骤2 → 步骤3 → ... → 输出
         ↓       ↓       ↓
      条件分支、循环、并行
```

---

### Type 3: 团队编排 Skill（Team Orchestration Skill）

**特点：** 协调多个智能体协作完成复杂任务

**示例：**
- `/team-combat` - 战斗团队编排（设计师+程序员+美术+音效+QA）
- `/team-ui` - UI 团队编排（UX+程序员+美术总监）
- `/team-narrative` - 叙事团队编排（叙事总监+作家+世界观设计师）

**结构：**
```
输入 → 分解任务 → 并行/串行委派给多个智能体 → 整合结果 → 输出
              ↓
         阶段检查点 + 用户决策
```

---

### Type 4: 元 Skill（Meta Skill）

**特点：** 关于 Skill 本身的 Skill（创建、管理、分析 Skill）

**示例：**
- `/skill-analyze` - 分析现有 Skill 的覆盖率和缺口
- `/skill-create` - 引导创建新的 Skill
- `/skill-optimize` - 优化现有 Skill 的性能

**结构：**
```
输入 → 分析/创建/优化 Skill 定义 → 验证 → 输出新的/改进的 Skill
```

---

## 📊 Skill 类型选择指南

| 场景 | 推荐类型 | 示例 |
|------|---------|------|
| 单一、明确的任务 | Type 1 | 生成变更日志、检查平衡性 |
| 多步骤标准化流程 | Type 2 | 原型开发、冲刺计划 |
| 需要多领域专家协作 | Type 3 | 实现战斗系统、设计UI |
| 创建/改进 Skill 本身 | Type 4 | 创建新 Skill、优化现有 Skill |

## ✅ 本课小结

### 核心概念

1. **Skill 是可复用的工作流程单元**，把复杂任务变成简单的 `/` 命令
2. **Skill 解决三个核心问题**：一致性、效率、知识沉淀
3. **四种 Skill 类型**：单一任务、工作流、团队编排、元 Skill

### 关键洞察

```
没有 Skill 的世界：
每次都要重新解释 → 结果不一致 → 质量不稳定 → 效率低下

有 Skill 的世界：
/命令触发 → 标准化执行 → 一致的高质量结果 → 高效复用
```

## 🙋 常见问题

**Q: Skill 和普通的提示词模板有什么区别？**  
A: Skill 是结构化的、可执行的、有状态的。它不仅仅是提示词，而是完整的工作流程定义，包括输入处理、多步骤执行、条件分支、输出生成。

**Q: 我需要为每个小任务都创建 Skill 吗？**  
A: 不需要。只有当任务满足以下条件时才值得创建 Skill：(1) 经常重复执行 (2) 需要一致的质量标准 (3) 涉及多个步骤容易遗漏 (4) 需要沉淀专家知识。

**Q: Skill 可以调用其他 Skill 吗？**  
A: 可以！这是高级用法。一个 Skill 可以在执行过程中调用其他 Skill，形成 Skill 链或 Skill 树，实现更复杂的工作流程。

**Q: 创建 Skill 需要编程基础吗？**  
A: 不需要传统编程。Skill 使用 YAML/Markdown 定义，重点是逻辑思考和流程设计。当然，有编程背景会更容易理解条件分支、循环等概念。
