## 3. 遇到的问题与解决方案

### 问题 1: UHT 无法识别 TSharedPtr

**错误信息:**
```
NodeGraphFormatter.h(25): error: Unable to find 'class', 'delegate', 'enum', or 'struct' with name 'TSharedPtr'
```

**原因:** UHT (Unreal Header Tool) 对 `TSharedPtr` 模板的支持有限，特别是在 USTRUCT 中使用时会失败。

**解决方案:**
1. 原始尝试: 添加 `#include "Templates/SharedPointer.h"` - 无效
2. 最终方案: 将 `TSharedPtr<FLayoutNode>` 改为 `FLayoutNode*` 原始指针，并移除 `UPROPERTY()` 标记

```cpp
// 修改前 (NodeGraphFormatter.h)
TArray<TSharedPtr<FLayoutNode>> InputConnections;
TArray<TSharedPtr<FLayoutNode>> OutputConnections;

// 修改后
TArray<FLayoutNode*> InputConnections;
TArray<FLayoutNode*> OutputConnections;
```

### 问题 2: UHT 不允许 USTRUCT 中暴露指向自身类型的指针

**错误信息:**
```
Inappropriate '*' on variable of type 'FLayoutNode', cannot have an exposed pointer to this type.
```

**原因:** UHT 限制在 USTRUCT 中使用指向自身类型的指针作为 UPROPERTY。

**解决方案:** 移除指针成员的 `UPROPERTY()` 标记，使其成为内部辅助成员（不需要序列化）

```cpp
USTRUCT(BlueprintType)
struct FLayoutNode
{
    GENERATED_BODY()

    UPROPERTY()
    UEdGraphNode* Node = nullptr;

    UPROPERTY()
    int32 Layer = 0;

    UPROPERTY()
    int32 IndexInLayer = 0;

    // 移除 UPROPERTY() - 这是内部辅助成员
    TArray<FLayoutNode*> InputConnections;
    TArray<FLayoutNode*> OutputConnections;
};
```

### 问题 3: FUIExtentenderBuilder 类型名拼写错误

**错误信息:**
```
error C2065: 'FUIExtentenderBuilder': undeclared identifier
```

**原因:** 原始代码中使用了不存在的类型 `FUIExtentenderBuilder`（拼写错误）。

**解决方案:** 移除未使用的函数声明

```cpp
// 修改前 (BlueprintNodeBridge.h)
private:
    TSharedPtr<FExtender> OnExtendLevelEditorMenu(const TSharedPtr<FUIExtentenderBuilder>& Builder);

// 修改后 - 移除该函数
private:
    void RegisterMenus();
    void UnregisterMenus();
```

### 问题 4: FAppStyle::Get().GetStyleSetName() 用法错误

**错误信息:**
```
error C2660: 'ISlateStyle::GetStyleSetName': function does not take 1 arguments
```

**原因:** UE 5.x 中 `FAppStyle::Get().GetStyleSetName()` 不接受参数。

**解决方案:**
```cpp
// 修改前 (BlueprintNodeBridgeCommands.h)
FAppStyle::Get().GetStyleSetName("EditorStyles")

// 修改后
FAppStyle::Get().GetStyleSetName()
```

### 问题 5: BlueprintEditor 模块依赖问题

**错误信息:**
```
Could not find definition for module 'BlueprintEditor', (referenced via Target -> BlueprintNodeBridge.Build.cs)
```

**原因:**
1. `BlueprintEditor` 是编辑器模块，在插件编译的规则阶段无法解析
2. 原始代码尝试将其添加到 `PublicDependencyModuleNames` 和 `DynamicallyLoadedModuleNames` 都失败

**解决方案:**
1. 不在 Build.cs 中声明依赖
2. 使用纯动态模块加载方式（运行时加载）
3. 简化代码，移除对 BlueprintEditor API 的直接依赖

```cpp
// 在 cpp 文件中动态加载
#include "Modules/ModuleManager.h"
#include "BlueprintEditorModule.h"

// 使用前检查
if (FModuleManager::Get().IsModuleLoaded("BlueprintEditor"))
{
    FBlueprintEditorModule& Module = FModuleManager::GetModuleChecked<FBlueprintEditorModule>(TEXT("BlueprintEditor"));
    // ...
}
```

### 问题 6: 委托签名不匹配

**错误信息:**
```
cannot convert 'void (__cdecl FBlueprintNodeBridgeModule::* )(UObject *)' to 'void (__cdecl FBlueprintNodeBridgeModule::* )(EBlueprintType) const'
```

**原因:** `OnBlueprintEditorOpened` 的委托签名与 `IBlueprintEditor::OnBlueprintEditorOpened()` 事件不匹配。

**解决方案:** 移除该回调函数，简化代码

## 6. 注意事项

1. **UE 版本差异**: UE 5.7 的 API 与早期版本有较大变化，特别是 BlueprintEditor 模块
2. **UHT 限制**: Unreal Header Tool 对模板类型和某些 C++ 特性的支持有限
3. **编辑器模块依赖**: 编辑器专用模块应在运行时加载，避免在 Build.cs 中声明依赖
4. **代码简化**: 遇到 API 不兼容时，优先简化代码，后续再逐步完善功能

## 9. 新增问题与解决方案

### 问题 7: EAutomationTestFlags 不可见

**错误**: `ApplicationContextMask` / `NormalContext` 未定义

**解决**: 使用自定义轻量级测试框架，避免 Automation Test 依赖

```cpp
#define CHECK(Condition, Msg) if (!(Condition)) { OutResults.Add(FTestResult(false, Msg)); }
```

### 问题 8: IConsoleManager.h 路径错误

**错误**: 无法打开包含文件 'IConsoleManager.h'

**解决**: 使用正确路径 `#include "HAL/IConsoleManager.h"`

### 问题 9: Console Command 委托签名不匹配

**错误**: CreateStatic 参数类型不匹配

**解决**: 移除 Console Command，改为在 StartupModule 直接调用测试

### 问题 10: 测试用例格式不匹配

**错误**: `Node1 -> Node2` 无括号导致解析失败

**解决**: 改用 `Node1 (outExec) -> Node2 (inExec)` 格式
